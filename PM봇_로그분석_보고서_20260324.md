## [6] 재발 방지를 위한 관찰 사항

> ⚠️ **이 섹션은 모니터링·알림 관점 관찰만 포함합니다. 코드 수정 제안 제외.**

### 모니터링 알림 설정 관점

1. **합성 반복 감지 알림**: 동일 parent_task_id에 대해 `_synthesize_and_act`가 N회(예: 3회) 이상 호출 시 Telegram 알림 발송 필요
2. **`needs_review` 상태 장기 체류 알림**: parent 태스크가 `needs_review`에 30분 이상 체류 시 경고
3. **LLM 호출 빈도 이상 감지**: 5분 내 동일 입력 해시로 10회 이상 `complete()` 호출 시 circuit breaker 알림
4. **API 비용 급증 감지**: 분당 Claude 세션 생성 수가 임계값(예: 5건/분) 초과 시 즉시 알림
5. **CONFLICTING 반복 감지**: 동일 parent에서 CONFLICTING 판단이 3회 이상 반복될 경우 자동 `done` 처리 + 알림

### 운영 절차 관점

6. **CONFLICTING 발생 시 수동 확인 프로세스**: `needs_review` 태스크 목록을 매일 09:00 KST에 Telegram 보고
7. **루프 발생 시 긴급 차단**: `pm_orchestrator` 재시작 없이 특정 parent_task를 강제 `done` 처리하는 운영 CLI 도구 필요
8. **개발·운영실 결과 상충 패턴**: T-362처럼 브랜치 미머지로 인한 상충은 별도 운영 게이트(머지 확인) 절차로 선행 처리
