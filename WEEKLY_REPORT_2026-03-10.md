# 프리즘 인사이트 주간 보고서

기간: 2026년 3월 4일 ~ 2026년 3월 10일

## 1. 이번 주 요약

이번 주 프리즘 인사이트는 `telegram-ai-org` 프로젝트의 핵심 구조를 단순 워커 라우팅 단계에서 한 단계 확장해, "동적 AI 조직"에 가까운 실행 기반을 확보했다. 특히 고정 역할 중심 구조에서 벗어나 요청별로 최적 에이전트 조합을 구성하는 방향으로 아키텍처를 전환했고, 설치/운영 경험과 메모리 계층, 워커 상태 추적, 테스트 안정성까지 함께 보강했다.

결과적으로 이번 주에는 다음 세 가지가 가장 큰 진전이었다.

- 동적 팀 빌딩 구조 도입: `workers.yaml` 중심의 정적 구성에서 `AgentCatalog` + `DynamicTeamBuilder` 기반 구조로 확장
- 운영 신뢰성 강화: 워커 헬스 모니터링, 프로젝트 메모리, 세션 관리, 설치 마법사 고도화
- 검증 기반 확보: 시뮬레이션 및 라우팅/완료/메모리 관련 테스트 확장, 전체 테스트 73건 통과

## 2. 주요 진행 사항

### 2-1. 아키텍처 고도화

- `~/.claude/agents/`에 정의된 21개 에이전트 페르소나를 재사용하는 `AgentCatalog`를 도입했다.
- 요청별로 필요한 역할과 수량, 실행 모드를 결정하는 `DynamicTeamBuilder`를 추가했다.
- 실행 방식은 `omc_team`, `agent_teams`, `sequential`의 3가지 모드로 분기되도록 설계했다.
- `workers.yaml`의 의미를 "실행 팀 정의"에서 "힌트 정보"로 축소해, 런타임 팀 구성이 더 유연해졌다.

### 2-2. PM 중심 실행 오케스트레이션 강화

- PM Bot이 단순 라우터를 넘어 팀 구성, 상태 추적, 완료 판정까지 총괄하는 구조로 정리됐다.
- TaskPlanner를 통해 순차/병렬 서브태스크 분해가 가능해졌고, 복합 요청에 대한 대응력이 개선됐다.
- Completion Protocol 기반으로 전체 태스크 완료 확인 흐름이 정교해졌다.

### 2-3. 메모리 및 운영 기반 개선

- Project Memory를 강화해 과거 태스크 이력과 맥락을 이후 라우팅 및 의사결정에 재활용할 수 있도록 했다.
- `CORE / SUMMARY / LOG` 3계층 메모리 구조와 중요도 기반 관리 로직이 추가됐다.
- Session Manager 및 tmux 세션 관리 로직이 포함되어 장기 실행 안정성이 개선됐다.
- Worker Health Monitor를 통해 워커 상태와 실행 가능성을 보다 체계적으로 추적할 수 있게 됐다.

### 2-4. 설치 및 운영 경험 개선

- `setup_wizard.py`가 preflight 점검, agent teams/omc 확인, 다중 조직 설정, simulation 검증까지 포괄하도록 대폭 확장됐다.
- 최신 커밋 기준으로 Privacy Mode 비활성화 필요 사항까지 설치 단계에서 명시하도록 보완했다.
- 실행 엔진 선택 항목이 설치 플로우에 반영되어 초기 설정 난도가 낮아졌다.

### 2-5. 멀티 조직 확장 기반 마련

- `organizations.yaml`, `OrgRegistry`, `CrossOrgBridge`가 추가되며 단일 팀을 넘어 복수 조직 구조로 확장할 수 있는 기반이 만들어졌다.
- 향후 조직별 PM Bot과 그룹 채팅방을 분리 운영하는 구조로 자연스럽게 이어질 수 있다.

## 3. 검증 현황

- `pytest -q` 기준 전체 테스트 73건이 통과했다.
- 검증 범위에는 메시지 스키마, PM 라우팅, 완료 프로토콜, 시뮬레이션, 프로젝트 메모리, 워커 헬스, 태스크 플래너가 포함된다.
- 현재 확인된 이슈는 기능 실패가 아니라 `datetime.utcnow()` 사용에 대한 deprecation warning 13건이며, 향후 timezone-aware 처리로 정리할 필요가 있다.

## 4. 이번 주 핵심 성과 해석

이번 주 성과의 의미는 기능 몇 개가 추가된 수준이 아니라, 프로젝트의 중심축이 "정적 봇 집합"에서 "요청 기반 동적 AI 팀"으로 이동했다는 점에 있다. 이는 프리즘 인사이트가 향후 다양한 업무 유형을 하나의 텔레그램 오피스 안에서 처리하는 방향으로 확장하는 데 필요한 구조적 전환이다.

특히 설치 경험, 메모리 구조, 상태 추적, 테스트 보강이 병행되었다는 점에서 단순 데모가 아니라 반복 실행 가능한 운영형 시스템으로 다가가고 있다.

## 5. 리스크 및 확인 사항

- `tools/claude_code_runner.py`에는 아직 커밋되지 않은 수정 사항이 1건 있어, 다음 배포 전 변경 의도 확인이 필요하다.
- `--permission-mode bypassPermissions` 기반 실행은 현재 내부 신뢰 환경에서는 빠른 검증에 유리하지만, 운영 환경 확대 시 보안 경계 재설계가 필요하다.
- 멀티 조직 구조와 동적 팀 구성은 설계가 앞서가고 있으므로, 실제 텔레그램 실사용 시나리오 기준의 E2E 검증이 다음 단계의 핵심이다.

## 6. 다음 주 제안

- 동적 팀 구성 결과를 실제 텔레그램 상호작용과 연결하는 E2E 검증 강화
- 멀티 조직 라우팅 시나리오의 실제 동작 테스트 추가
- `datetime.utcnow()` 경고 해소 및 운영 로그 타임존 일관성 정비
- 보안 경계 점검: 실행 권한 모델, 토큰 관리, 샌드박스 전략 구체화
- 핵심 사용자 데모 시나리오 2~3개를 정의해 PM Bot 중심 스토리라인 완성

## 7. 한 줄 결론

이번 주 프리즘 인사이트는 "텔레그램 기반 AI 조직"이라는 비전을 설명하는 수준을 넘어, 동적 팀 구성과 운영 기반을 갖춘 실행 가능한 프로토타입 단계로 진입했다.
