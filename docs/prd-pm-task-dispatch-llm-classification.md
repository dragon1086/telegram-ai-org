# PRD: PM Task Dispatch — LLM 자율 태스크 분류 설계

> 작성: aiorg_product_bot (PM)
> 날짜: 2026-03-23
> 태스크: T-aiorg_pm_bot-346

---

## 배경

### 문제 정의
현재 PM task dispatch 배분 메시지는 태스크 유형을 단 3가지(분석/보고서, 구현/파일생성, 혼합)로만 구분한다.
실제 실무에서는 "분석해줘"와 "조사해줘"가 전혀 다른 산출물을 요구하고, "기획해줘"와 "설계해줘"도 범위가 다름에도, LLM은 이를 구분하는 기준을 갖지 못했다.

### 발생한 오버스콥 사례 (Phase 1 분석)
| 사례 | 원 요청 유형 | 실제 수행 | 오분류 방향 |
|------|------------|---------|-----------|
| 리서치실 TradingAgents 조사 | 조사(보고서 1개) | 중간 메모 13개 파일 추가 생성 | 조사 → 구현(파일생성) |
| 성장실 경쟁분석 Phase 1 | 분석 | Phase 2 포지셔닝 전략까지 자동 실행 | 분석 → 기획 확장 |
| PM 봇 약점 분석 | 분석(보고서) | settings.json 편집 코드·YAML 스니펫 포함 | 분석 → 구현 사양서 |
| 개발실 "봐줘" 요청 | 검토/분석 | pm_orchestrator.py 등 실제 코드 수정 | 검토 → 구현 |

### 구조적 원인
- LLM에게 "왜 이 유형인지" 판단 근거를 요구하는 장치가 없었다
- "혼합" 유형이 사실상 범위 무제한 허용으로 작동했다
- 배분 메시지에 "파일 변경 여부"를 명시하지 않아 봇이 임의 판단

---

## 목표

1. LLM이 태스크 유형을 스스로 판단·분류하도록 유도
2. 배분 메시지에 유형 + 판단 근거 + 파일 변경 허용 여부를 명시해 오버스콥을 구조적으로 차단
3. 오분류 시에도 안전한 방향(사고형 우선)으로 fallback되도록 설계

---

## 기능 요구사항

### FR-1: 태스크 유형 8종 분류 체계

LLM은 배분 전 아래 8종 중 하나로 태스크를 분류해야 한다.

| # | 유형 | 파일 변경 | 핵심 뉘앙스 |
|---|------|-----------|------------|
| 1 | 조사 | 아니오 | 사실 수집, 레퍼런스 탐색. 해석 없음 |
| 2 | 분석 | 아니오 | 데이터 해석, 비교, 인사이트 추출 |
| 3 | 기획 | 아니오 | 무엇을 만들지 결정. PRD·요구사항 산출 |
| 4 | 설계 | 아니오 | 어떻게 만들지 구체화. 다이어그램·명세 |
| 5 | 검토 | 아니오 | 기존 산출물 품질 검증. 의견 제시 |
| 6 | 수정 | 예 | 기존 것 일부 변경. 범위 엄격 제한 |
| 7 | 구현 | 예 | 새로 만들기. 코드·파일 생성 |
| 8 | 운영 | 예 | 인프라·배포·재시작. infra 봇 전용 |

### FR-2: 판단 체인 (Chain-of-Thought 유도)

배분 메시지 작성 전 Step 0에서 다음 4개 질문을 순서대로 답해야 한다:
1. 최종 산출물이 코드/파일인가 vs 문서/보고서인가?
2. 핵심 동사가 무엇인가?
3. 파일·코드를 직접 변경해야 하는가?
4. 유형 확신도가 60% 이상인가?

### FR-3: 배분 메시지 신규 필수 필드

기존 배분 메시지에 아래 2개 필드 추가 (필수):
- `태스크 유형:` — 8종 중 하나
- `판단 근거:` — 핵심 동사/의도 한 줄 요약
- `파일·코드 변경 허용:` — 예 / 아니오

### FR-4: Fallback 정책

확신도 < 60% 시 적용:
1. 단계 분리 배분: 조사 → 분석 → 기획 → 구현 순서로 단계별 배분
2. 사고형 우선: 불확실하면 "분석"(기본값)으로 먼저 배분, 산출물 검토 후 다음 단계 결정
3. "혼합이니까 다 해도 된다" 해석 금지

### FR-5: 오버스콥 검수 기준

`파일·코드 변경 허용: 아니오`인 태스크에서 봇이 파일을 생성/수정/삭제한 경우:
- PM은 해당 결과물을 오버스콥으로 판정
- 파일 변경사항은 별도 승인 태스크로 분리

---

## 비기능 요구사항

- **NFR-1 간결성**: 판단 근거는 1줄(50자 이내). 장문 설명 금지
- **NFR-2 일관성**: 동일한 동사 신호는 항상 같은 유형으로 분류 (프롬프트 내 테이블 고정)
- **NFR-3 자기설명**: LLM이 유형을 잘못 선택한 경우, 판단 근거 필드를 보고 PM이 즉시 오분류를 감지 가능
- **NFR-4 범용성**: 새로운 태스크 유형이 등장해도 8종 분류표의 "가장 가까운 유형"으로 흡수 가능해야 함

---

## 제약사항

- 구현(코드 변경)은 PM 또는 사용자의 명시적 승인 없이 착수 금지
- "혼합" 유형 제거 — 단계 분리 배분으로 대체
- 유형 판단 로직을 별도 코드로 구현하지 않음 — 프롬프트(SKILL.md) 수준에서만 작동

---

## 성공 지표

| 지표 | 현재 | 목표 |
|------|------|------|
| 사고형 요청에서 파일 변경 발생률 | 측정 불가 (오분류 감지 장치 없음) | < 5% |
| 배분 메시지에 태스크 유형 명시율 | 0% (필드 없음) | 100% |
| 오버스콥 판정 평균 탐지 시간 | 사후 발견 | 배분 메시지 검토 시 즉시 |
| 분석 요청이 구현으로 확장되는 빈도 | 4건/10태스크 (관찰치) | < 1건/10태스크 |

## Few-shot 배분 메시지 예시

### 예시 1 — 조사 요청

```
사용자: "TradingAgents 관련 오픈소스 프레임워크 조사해줘"

Step 0 판단:
- Q1: 산출물 = 보고서 → 사고형
- Q2: 핵심 동사 = "조사해줘" → 조사
- Q3: 파일 변경 필요? 아니오
- Q4: 확신도 95%

[태스크 배분] #T-001
담당: @research_bot
태스크 유형: 조사
판단 근거: "조사해줘" 동사 + 레퍼런스 수집이 목적, 해석 불필요
파일·코드 변경 허용: 아니오
내용: TradingAgents 관련 오픈소스 멀티에이전트 트레이딩 프레임워크 목록 수집
긴급도: 오늘중
기대 산출물: 프레임워크 목록 + 각 프레임워크 개요 1-2줄 (보고서 형식)
실행 범위: 조사·보고서 작성만. 코드 분석·구현 제안 포함 금지
```

### 예시 2 — 분석 요청 (오버스콥 위험 사례)

```
사용자: "단일 에이전트 vs 멀티에이전트 약점 분석해줘"

Step 0 판단:
- Q1: 산출물 = 분석 보고서 → 사고형
- Q2: 핵심 동사 = "분석해줘" → 분석
- Q3: 파일 변경 필요? 아니오
- Q4: 확신도 90% (주의: 분석 결과로 구현 제안이 나올 수 있음 → 실행 범위에 명시)

[태스크 배분] #T-002
담당: @research_bot
태스크 유형: 분석
판단 근거: "분석해줘" 동사 + 비교 해석이 목적, 코드 변경 불필요
파일·코드 변경 허용: 아니오
내용: 단일 에이전트 대비 멀티에이전트 약점 비교 분석
긴급도: 이번주
기대 산출물: 비교 분석 보고서 (표 + 결론 포함)
실행 범위: 분석 보고서 작성만. settings.json·YAML·코드 스니펫 삽입 금지
```

### 예시 3 — 구현 요청

```
사용자: "dispatch_engine.py에 태스크 유형 필드 추가해줘"

Step 0 판단:
- Q1: 산출물 = 코드 변경 → 실행형
- Q2: 핵심 동사 = "추가해줘" → 구현/수정
- Q3: 파일 변경 필요? 예
- Q4: 확신도 95%

[태스크 배분] #T-003
담당: @engineering_bot
태스크 유형: 수정
판단 근거: 기존 파일(dispatch_engine.py) 일부 변경, 새 파일 생성 없음
파일·코드 변경 허용: 예
내용: dispatch_engine.py TaskDispatch 클래스에 task_type 필드 추가
긴급도: 이번주
기대 산출물: 수정된 dispatch_engine.py + 관련 테스트 통과
실행 범위: dispatch_engine.py 수정 + 관련 테스트만. 타 파일 수정 금지, 배포 금지
```
