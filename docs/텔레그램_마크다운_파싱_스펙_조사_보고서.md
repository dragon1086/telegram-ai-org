## 1. 핵심 결론 (Executive Summary)

**현재 코드베이스는 올바른 방향(HTML parse_mode)으로 구현되어 있다.**
`parse_mode="HTML"` + `markdown_to_html()` 컨버터 패턴이 일관되게 적용되어 있으며,
MarkdownV2나 레거시 Markdown은 단 한 건도 사용되지 않는다.

**그럼에도 렌더링이 안 된다면, 원인은 아키텍처가 아니라 아래 4개 리스크 지점 중 하나다.**

## 2. 현황 요약

### 2-1. parse_mode 분포

| parse_mode | 건수 | 비율 |
|------------|------|------|
| `"HTML"` | 80건+ | **100%** |
| `"MarkdownV2"` | 0건 | 0% |
| `"Markdown"` (레거시) | 0건 | 0% |
| 미지정 | 45건+ | — (단순 평문 알림) |

### 2-2. markdown_to_html() 커버리지

| 경로 | 적용 여부 |
|------|-----------|
| LLM 응답 → `_pm_send_message()` → `display.send_to_chat()` | ✅ 100% 적용 |
| 협업 결과 → `reply_text()` | ✅ 적용 |
| 진행 상황 업데이트 → `edit_text()` | ✅ 적용 |
| scripts/ 스케줄 메시지 | ✅ 전체 적용 |
| 수동 HTML f-string 메시지 (명령어 응답) | ⚠️ 변환 불필요 (직접 HTML 작성) |
| 단순 텍스트 알림 (parse_mode 미지정) | ✅ 불필요 (포매팅 없음) |

### 2-3. markdown_to_html() 지원 변환 목록

| 마크다운 | → HTML | 상태 |
|----------|--------|------|
| `**text**` / `__text__` | `<b>text</b>` | ✅ |
| `*text*` / `_text_` | `<i>text</i>` | ✅ |
| `***text***` / `___text___` | `<b><i>text</i></b>` | ✅ |
| `` `code` `` | `<code>code</code>` | ✅ |
| ` ```...``` ` | `<pre>...</pre>` | ✅ |
| `# H1` | `<b>H1</b>\n──────────` | ✅ |
| `## H2` | `<b>▸ H2</b>` | ✅ |
| `### H3~H6` | `<b>H3</b>` | ✅ |
| `[text](url)` | `<a href="url">text</a>` | ✅ |
| `~~text~~` | `<s>text</s>` | ✅ |
| `> text` | `<blockquote>text</blockquote>` | ✅ |
| `- item` / `+ item` | `• item` | ✅ |
| `---` / `***` / `___` | `──────────` | ✅ |
| 테이블 행 | 셀 구분자 변환 | ✅ |
| `&` `<` `>` | `&amp;` `&lt;` `&gt;` | ✅ |

## 3. 리스크 항목

### 🔴 Risk-1: 수동 HTML f-string에 동적 데이터 이스케이프 누락

**해당 위치**: `core/telegram_relay.py` 3640~3686, 3807~3812 일대

```python
# 예시 — 문제가 되는 패턴
msg = (
    f"• 방향성: {d.get('direction','미설정')}\n\n"  # ← 이스케이프 없음!
    f"<code>/org@{bot_name} ...</code>\n\n"
)
await update.message.reply_text(msg, parse_mode="HTML")
```

- `d.get('direction')` 값에 `&`, `<`, `>` 포함 시 HTML 파싱 오류
- 텔레그램은 HTML 태그가 깨진 메시지에서 400 Bad Request 반환하거나 태그를 무시
- **수정 방법**: `escape_html()` 래핑 필요

```python
from core.telegram_formatting import escape_html
msg = (
    f"• 방향성: {escape_html(d.get('direction','미설정'))}\n\n"
    f"<code>/org@{escape_html(bot_name)} ...</code>\n\n"
)
```

### 🟡 Risk-2: `* item` 형식 bullet 목록 미지원

**문제**: `markdown_to_html()`은 `- item`과 `+ item`을 `•`로 변환하지만, `* item` (별표 bullet)은 이탤릭(`<i>item</i>`)으로 잘못 변환될 수 있다.

```python
# 현재 코드 (telegram_formatting.py line 120)
text = re.sub(r"\*([^*\n]+?)\*", r"<i>\1</i>", text)  # * 하나를 이탤릭으로 처리
# → "* 항목"의 경우 "항목" 앞 공백이 있어 실제 매칭 여부는 컨텍스트에 따라 다름
```

- LLM이 `* item` 형식으로 목록을 생성하면 렌더링 불일치 발생 가능
- **수정 방법**: bullet 변환 정규식에 `*` 추가 또는 시스템 프롬프트에서 `- item`만 사용 강제

### 🟡 Risk-3: 중첩 마크다운에서 DOTALL 플래그 오작동

```python
# telegram_formatting.py line 117
text = re.sub(r"\*\*(.+?)\*\*", r"<b>\1</b>", text, flags=re.DOTALL)
```

`re.DOTALL` 사용으로 `**`가 여러 줄에 걸쳐 greedy 매칭될 수 있음.
LLM이 미완성 bold syntax(`**` 미닫힘)를 출력하면 다음 `**`까지 전체가 bold 처리될 위험.

### 🟢 Risk-4: 채널에서의 `<blockquote>` 지원 여부

- `<blockquote>`는 Bot API 6.0+ (2022년 이후)에서 지원됨
- 구형 클라이언트나 일부 채널 환경에서 렌더링이 무시될 수 있음
- 현재 코드는 이를 올바르게 사용 중이나, 구형 클라이언트 사용자에겐 평문으로 표시될 수 있음

## 4. 권고 조치

### 우선순위 1 (즉시 수정 권장)

**수동 HTML f-string의 동적 데이터 이스케이프 추가**

```python
# core/telegram_relay.py 수정 대상 지점들
# 3648~3658, 3679~3685, 3807~3811, 4085~4091 등

# 패턴: d.get('xxx') → escape_html(d.get('xxx', ''))
```

### 우선순위 2 (중기 개선)

**`* item` bullet 변환 추가**:
```python
# telegram_formatting.py에 아래 줄 추가 (이탤릭 변환 전에 처리)
text = re.sub(r"^([ \t]*)\* (?!\*)", r"\1• ", text, flags=re.MULTILINE)
```

### 우선순위 3 (검토)

**`re.DOTALL` 플래그 제거 및 단일 줄 내 bold 변환만 허용**:
```python
# 변경 전
text = re.sub(r"\*\*(.+?)\*\*", r"<b>\1</b>", text, flags=re.DOTALL)
# 변경 후 — 줄 내에서만 매칭
text = re.sub(r"\*\*([^\n]+?)\*\*", r"<b>\1</b>", text)
```

## 5. parse_mode 표준 권고: HTML 유지

MarkdownV2 vs HTML 중 **HTML이 현재 코드베이스에 더 적합**한 이유:

1. **LLM 출력 안전성**: LLM은 `-`, `.`, `_`, `!` 등 MarkdownV2 이스케이프 대상 문자를 수시로 출력함 → MarkdownV2 사용 시 전체 텍스트 이스케이퍼 별도 구현 필요
2. **기존 converter 재사용**: `markdown_to_html()`은 이미 검증된 로직이며 80개 이상의 send 호출에서 사용 중
3. **이스케이프 단순성**: HTML은 `&`, `<`, `>` 3개만 처리하면 됨
4. **태그 호환성**: `<b>`, `<i>`, `<code>`, `<pre>`, `<blockquote>`, `<a>` 등 텔레그램 HTML 지원 태그가 충분히 풍부함
