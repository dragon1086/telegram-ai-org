### 시나리오 SCN-01: 단순 코딩 태스크 (Simple Coding)

**목적**: 단일 기능 수정 태스크에서의 속도·품질 비교. 단일 세션이 유리한 구간이 어디까지인지 파악.

**입력 조건**

```
[표준화 지시문]
"core/shared_memory.py의 get() 메서드에 TTL(Time-to-Live) 파라미터를 추가하라.
- 기존 호출부(기본값=None)와 하위 호환 유지
- set() 메서드와 TTL 연동
- 단위 테스트 3개 이상 (정상 케이스 2 + 경계 케이스 1)
- ruff 린트 통과 필수"
```

**동일 입력 적용 범위**
- 단일 세션: 새 Claude Code 세션에서 위 지시문 그대로 입력
- 멀티에이전트: `aiorg_engineering_bot` 수신 태스크로 위 지시문 전달

**측정 KPI 및 목표**

| KPI | 단일 세션 기준선 | 멀티에이전트 목표 | 최소 우위 임계값 |
|-----|-------------|---------------|--------------|
| TTD-Simple | 5-10분 | ≤8분 | 단일 세션과 동등 이상 |
| QS-Accuracy | 70-85점 | ≥85점 | 동등 또는 우위 |
| ERR-Detection | 0% | ≥80% | 자동 감지 확보 |
| ERR-Recovery | 0% | ≥70% | 자동 복구 확보 |

> **해석 기준**: SCN-01은 단일 세션이 유리한 구간. 우리 시스템이 여기서도 "동등 이상"이어야 한다. 속도가 느려지면 훅 오버헤드 또는 에이전트 라우팅 지연이 원인.

**반복 횟수**: n=5 (동일 지시문, 각 실행 전 상태 초기화)

**실행 절차**:
1. 단일 세션 5회 실행 → `logs/baseline_experiments/exp-A-simple-coding/`
2. 멀티에이전트 5회 실행 → `logs/multiagent_experiments/exp-A-simple-coding/`
3. 쌍대 t-검정으로 통계적 유의성 확인 (p < 0.05 기준)

### 시나리오 SCN-02: 멀티팀 조율 태스크 (Multi-team Coordination)

**목적**: 여러 도메인(개발+운영+기획)에 걸친 태스크에서 멀티에이전트의 압도적 우위를 검증.

**입력 조건**

```
[표준화 지시문]
"신규 모니터링 봇 'health-monitor-bot'을 설계 및 추가하라.
- bots/health_monitor_bot.yaml 생성 (YAML 스키마 준수)
- orchestration.yaml에 봇 등록
- scripts/hooks/ 에 health-check.sh 생성 (봇 상태 5가지 체크)
- docs/에 health-monitor-runbook.md 작성
- tests/test_health_monitor.py (기본 테스트 3개)
모든 파일이 일관된 설계를 유지해야 하며 validate-config가 PASS여야 한다."
```

**핵심 설계 의도**: 이 태스크는 개발(YAML+코드) + 운영(훅) + 기획(문서) + 품질(테스트) 4개 도메인에 걸쳐 있다. 단일 세션은 순차 실행만 가능하며 도메인 간 일관성 유지에 취약하다.

**측정 KPI 및 목표**

| KPI | 단일 세션 기준선 | 멀티에이전트 목표 | 최소 우위 임계값 |
|-----|-------------|---------------|--------------|
| TTD-Multi | 45-90분 | ≤36분 | ≥20% 단축 |
| QS-Accuracy | 60-75점 | ≥85점 | +15점 이상 |
| QS-Consistency | 55-70점 | ≥85점 | +15점 이상 |
| PPE-MaxParallel | 1 | ≥3 | 병렬 처리 실증 |

**반복 횟수**: n=5

**실행 절차**:
1. 단일 세션: 모든 파일을 순차적으로 생성 → 완료 시각 기록
2. 멀티에이전트: PM → engineering/ops/research/product 동시 위임 → 각 봇 완료 후 통합
3. 도메인 간 일관성(QS-Consistency)은 4개 파일 상호 참조 정확도로 측정

### 시나리오 SCN-03: 장기 프로젝트 관리 (Long-term Project)

**목적**: 세션 경계를 넘는 장기 태스크에서 컨텍스트 보존·연속성 면의 압도적 우위 검증.

**입력 조건**

```
[표준화 지시문 — Phase 1]
"봇 성능 분석 파이프라인을 3단계로 구축한다.
Phase 1 (지금 실행):
- 데이터 수집 설계서 작성 (KPI 5개, 수집 방법, 저장 스키마)
- core/metrics_collector.py 스텁 구현 (인터페이스만, 로직 없음)
Phase 1 완료 후 새 세션에서 Phase 2 지시문을 전달한다."

[표준화 지시문 — Phase 2 (새 세션/24시간 후)]
"Phase 1에서 설계한 KPI 수집 파이프라인의 집계 로직을 구현하라.
- core/metrics_collector.py 로직 완성 (Phase 1 스텁 기반)
- core/metrics_aggregator.py 신규 작성
- Phase 1 설계서와 인터페이스 100% 일치 필수"

[표준화 지시문 — Phase 3 (새 세션/48시간 후)]
"Phase 1-2에서 구축한 메트릭 파이프라인으로 일일 보고서를 생성하라.
- reports/daily_metrics_report.py 구현
- 생성된 보고서가 Phase 1 KPI 정의와 일치해야 함"
```

**측정 KPI 및 목표**

| KPI | 단일 세션 기준선 | 멀티에이전트 목표 | 최소 우위 임계값 |
|-----|-------------|---------------|--------------|
| TTD-Long (시간) | 4-8시간 | ≤2.4시간 | ≥40% 단축 |
| CTX-CrossSession (%) | 20-40% | ≥80% | +40%p 이상 |
| CTX-LongTask (%) | 30-50% | ≥85% | +35%p 이상 |
| QS-Consistency (Phase간) | 45-65점 | ≥85점 | +20점 이상 |

**반복 횟수**: n=5 (각 실험은 3 Phase 전체 완주 = 1회로 계산)

**컨텍스트 보존 평가 방법**:
- Phase 1에서 정의한 KPI 5개의 이름·단위·측정 방법을 기록
- Phase 2, 3에서 동일 항목을 올바르게 참조했는지 항목별 0/1 평가
- CTX = 올바른 참조 수 / 15 (5개 × 3번 참조 기회)

## 2. 성능 우위 판단 임계값 기준표

### 2-1. KPI별 최소 우위 임계값

> 아래 임계값은 팀 합의 기준. 근거는 섹션 2-2에 명시.

| KPI | 단위 | 단일 세션 기준선 | 최소 우위 임계값 | 목표값 | 판정 |
|-----|------|-------------|--------------|--------|------|
| TTD-Simple | 분 | 7.5 (예상 중앙값) | -20% ≤ 6분 | ≤5분 | 동등 이상 확보 |
| TTD-Multi | 분 | 67.5 (예상 중앙값) | -20% ≤ 54분 | ≤36분 | 우위 필수 |
| TTD-Long | 시간 | 6 (예상 중앙값) | -30% ≤ 4.2시간 | ≤2.4시간 | 압도적 우위 |
| QS 종합 | 점 | 72 (예상 중앙값) | +15점 ≥ 87점 | ≥90점 | 우위 필수 |
| QS-Accuracy | 점 | 77 | +10점 ≥ 87점 | ≥90점 | 우위 필수 |
| QS-Consistency | 점 | 62 | +20점 ≥ 82점 | ≥88점 | 압도적 우위 |
| PPE-MaxParallel | 개 | 1 | ≥3 | ≥4 | 구조적 우위 |
| PPE-TimeReduction | % | 0% | ≥25% | ≥40% | 우위 필수 |
| CTX-CrossSession | % | 30 | +40%p ≥ 70% | ≥80% | 압도적 우위 |
| CTX-LongTask | % | 40 | +35%p ≥ 75% | ≥85% | 압도적 우위 |
| ERR-Detection | % | 0 | ≥70% | ≥85% | 압도적 우위 |
| ERR-Recovery | % | 0 | ≥60% | ≥75% | 압도적 우위 |
| ERR-MTTR | 분 | 20 | -50% ≤ 10분 | ≤5분 | 우위 필수 |

### 2-2. 임계값 근거

**TTD 임계값 (≥20% 단축)**
- 근거: 멀티에이전트 시스템의 라우팅 오버헤드(평균 1-3분)를 감안하면 20% 미만 단축은 실질적 이득이 없다. 단순 태스크에서 동등, 복잡 태스크에서 30-50% 이상이 압도적 우위 조건.

**QS 임계값 (+15점)**
- 근거: 평가자 간 신뢰도(inter-rater reliability) 오차 범위가 ±10점. 따라서 15점 이상 차이가 나야 시스템 차이로 판단 가능.
- 100점 기준에서 15점은 "한 단계 등급 상승(예: B→A)"에 해당하는 실질적 차이.

**PPE 임계값 (≥3개 동시 처리)**
- 근거: 6개 봇 중 최소 절반(3개)이 동시 활성 상태일 때 진정한 멀티에이전트 이점 발생. 1-2개는 순차 실행 대비 의미 있는 차이가 없음.

**CTX 임계값 (+40%p)**
- 근거: 단일 세션의 컨텍스트 창 한계로 30-40% 보존은 "운에 의존". 70% 이상이어야 "신뢰할 수 있는 컨텍스트 보존"으로 판단.

**ERR 임계값 (Detection ≥70%, Recovery ≥60%)**
- 근거: 단일 세션 0% 대비 어떤 수치든 개선이나, 실용성 기준:
  - 70% 미만 감지는 "없는 것과 비슷한" 수준
  - 60% 미만 복구는 사용자 신뢰 확보 불가
- 이 임계값은 훅 시스템(GAP-1) 해소 후에야 달성 가능

### 2-3. 종합 판정 기준

| 종합 점수 비율 | 판정 | 조치 |
|-------------|------|------|
| 멀티에이전트 ≥ 단일세션 × 1.50 | ✅ 압도적 우위 | 현 상태 유지 + 확장 |
| 멀티에이전트 ≥ 단일세션 × 1.30 | ✅ 우위 확보 | 약점 KPI 집중 개선 |
| 단일세션 × 1.15 ≤ 멀티 < × 1.30 | ⚠️ 부분 우위 | 개선 가속화 |
| 멀티에이전트 < 단일세션 × 1.15 | ❌ 우위 미달 | 긴급 개선 플랜 가동 |
| 멀티에이전트 < 단일세션 | 🚨 역전 (회귀) | 즉시 회귀 탐지 절차 실행 |

## 3. 비교 실험 수행 절차서

### 3-1. 실험 사전 준비 체크리스트

```markdown
[ ] 실험 디렉토리 생성: logs/baseline_experiments/, logs/multiagent_experiments/
[ ] git 브랜치 생성: experiment/baseline-YYYYMMDD
[ ] ai_org.db 스냅샷 백업: cp ai_org.db ai_org.db.backup-{날짜}
[ ] 봇 프로세스 상태 확인: ps aux | grep aiorg
[ ] 품질 게이트 기준 확인: python tools/orchestration_cli.py validate-config
[ ] 실험 로그 템플릿 복사: cp docs/benchmarks/templates/experiment-log.md logs/.../
```

### 3-2. 단일 세션 실험 실행 절차

```
Step 1: 환경 초기화
  - 새 Claude Code 세션 시작 (claude --no-resume)
  - cd /Users/rocky/telegram-ai-org
  - git checkout -b experiment/baseline-{시나리오}-{n}
  - 시작 시각 기록 (T_수신 = 현재 시각)

Step 2: 지시문 입력
  - 해당 시나리오의 "표준화 지시문" 그대로 붙여넣기
  - 추가 힌트나 수정 지시 없음 (오류 시 1회 "힌트 없이 재시도" 허용)
  - 사용자 추가 입력 없이 Claude가 완료 선언할 때까지 대기

Step 3: 완료 기록
  - 완료 선언 시각 기록 (T_완료)
  - TTD = T_완료 - T_수신
  - git diff로 변경 파일 목록 기록

Step 4: 산출물 평가
  - AC 체크리스트 수동 평가 (각 항목 0/1)
  - QS-Accuracy = (AC 충족 수 / 전체 AC 수) × 100
  - ruff + pytest 실행: python -m ruff check . && .venv/bin/pytest -q --tb=short
  - 오류 발생 횟수·유형 기록

Step 5: 결과 저장
  - logs/baseline_experiments/exp-{시나리오}-run-{n}.md 파일 작성
  - git checkout main && git branch -d experiment/...
```

### 3-3. 멀티에이전트 실험 실행 절차

```
Step 1: 환경 초기화
  - ai_org.db 초기화 확인 (미완료 태스크 없음)
  - 6개 봇 프로세스 확인 및 재시작 (request_restart.sh)
  - 시작 시각 기록 (T_수신)

Step 2: 태스크 투입
  - 텔레그램 또는 직접 PM 봇에 표준화 지시문 전달
  - PM 봇의 팀 구성 및 위임 과정 로그 기록
  - 봇 간 P2PMessenger 메시지 타임스탬프 기록

Step 3: 완료 확인
  - 마지막 봇 완료 선언 시각 기록 (T_완료)
  - TTD = T_완료 - T_수신

Step 4: 산출물 평가
  - 단일 세션과 동일한 AC 체크리스트 적용
  - ruff + pytest 실행 (동일 조건)
  - 병렬 처리 수: max(동시 활성 봇 수) 기록
  - 오류 자동 감지/복구 횟수 로그에서 추출

Step 5: 결과 저장
  - logs/multiagent_experiments/exp-{시나리오}-run-{n}.md 작성
```

### 3-4. 비교 분석 절차

```
Step 1: 데이터 수집 완료 확인
  - 각 시나리오별 단일 세션 n=5, 멀티에이전트 n=5 완료 확인

Step 2: 통계 계산
  - 각 KPI별 평균(μ), 표준편차(σ), 중앙값 계산
  - 쌍대 t-검정 (paired t-test): p < 0.05 → 통계적 유의미
  - 효과 크기(Cohen's d): d > 0.5 → 실질적 차이

Step 3: 임계값 판정
  - 섹션 2-1의 기준표와 실측값 비교
  - 항목별 PASS/FAIL 판정

Step 4: 종합 보고서 작성
  - docs/benchmarks/official-benchmark.md 업데이트
  - 미달 KPI 원인 분석 및 개선 과제 도출
```

### 3-5. 실험 반복 원칙

| 원칙 | 내용 |
|------|------|
| 최소 반복 횟수 | n≥5 (통계적 유의성 확보 기준) |
| 실험 간격 | 단일 시나리오 내 반복 간 최소 30분 (컨텍스트 초기화 보장) |
| 이상치 처리 | IQR 방식으로 이상치 제거 후 재계산 (n≥4 유지) |
| 블라인드 평가 | QS 수동 평가는 단일/멀티 구분 없이 산출물 코드만 보고 평가 |
| 기록 의무 | 모든 실험 결과는 로그 파일로 저장 (비공개 보관) |
