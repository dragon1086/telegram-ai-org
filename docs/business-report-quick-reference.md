# 비즈니스 보고서 Quick Reference

> 1페이지 요약본 | 2026-03-23 | 전체 가이드: `docs/business-report-guideline.md`

---

## 보고서 = 결론 먼저, 근거는 뒤에 (피라미드 원칙)

```
[LEVEL 1] 결론/권고        → 첫 문단, 30초 내 파악
[LEVEL 2] 핵심 근거 3~5개 → 본문, 2~3분 내 파악
[LEVEL 3] 상세 데이터      → 부록, 필요 시만 참고
```

---

## Executive Summary 공식 (CQC + BLUF)

```
첫 줄: [결론] + [PM 권고]           ← BLUF
본문: ① 왜 이 보고인가 (Context)
      ② 핵심 질문 3~5개 (Questions)
      ③ 각 질문의 답 (Conclusions)
길이: 최대 10줄 / 텔레그램 기준
```

---

## 7-Point 자가 점검 체크리스트

```
[ ] 1. 첫 3줄 = 결론
[ ] 2. Executive Summary = 1페이지 이내
[ ] 3. 중복 섹션 없음 (MECE)
[ ] 4. 헤더 = 결론 담은 명사구
[ ] 5. 팀 원문 bypass 없음 (PM 재작성)
[ ] 6. PM 권고 명시
[ ] 7. 상세 내용 = 부록 분리
```

**5개 미만 통과 → 발송 금지, 구조 재설계**

---

## MECE 체크 (중복·누락 제거)

| 같은 이슈가 2팀에서 나오면 | → 1개로 통합 |
|팀 이름으로 섹션 나누면 | → 주제(이슈/근거/권고)로 재구성 |
| 빠진 관점 있으면 | → PM이 직접 추가 |

---

## 헤더 작성 원칙

| ❌ 빈 헤더 | ✅ 결론 헤더 |
|-----------|------------|
| 현황 | 배포 자동화 부재가 핵심 병목 |
| 이슈 | CI/CD 미구축으로 배포당 3시간 소요 |
| 검토 결과 | 3팀 분석 결과: 우선순위 1위는 인프라 |

---

## 가독성 5원칙

1. **한 문단 = 최대 3~4줄**
2. **핵심 수치는 **굵게** + 단위 포함**
3. **비교 정보는 표(table)로**
4. **섹션 구분은 `---` 수평선으로**
5. **팀별 원문은 부록(Appendix)으로**
