# PM 최종보고 개선안 통합 PRD v1.0

> 기준일: 2026-03-23 | 작성: aiorg_product_bot
> 목적: 다부서 취합 후 PM 최종보고의 품질 기준·포맷·운영 지침을 단일 문서로 통합

---

## 1. 문제 정의 (Why)

### 현행 보고 흐름
```
사용자 요청
→ PM 분해 → 부서별 실행
→ ResultSynthesizer.synthesize()   ← LLM 1차 합성
→ ensure_user_friendly_output()    ← LLM 2차 재작성
→ Telegram 전송
```

### 식별된 결함 (3가지)

| # | 결함 | 원인 | 영향 |
|---|------|------|------|
| 1 | 바이패싱 | 팀 결과를 정제 없이 통과 | 중복·비정제 보고 |
| 2 | 중복 | MECE 클러스터링 단계 부재 | 같은 내용 2~3번 반복 |
| 3 | 비정제 | PM 관점 재구성 규칙 약함 | 팀별 원문 copy-paste 형태 |

### 추가 기술적 문제

- `_SYNTHESIS_PROMPT`: 섹션 4개뿐, MECE 지시 없음, PM 권고 섹션 없음
- `ensure_user_friendly_output`: 3섹션으로 재작성 → 구조 불일치
- `original_request` 500자 잘림 → 문맥 손실
- "판단 근거" 섹션이 "핵심 내용"과 중복됨

## 2. 취합·정제·중복제거 원칙 명세 (Phase 1 산출물)

### 2-1. 취합 내용 반영 원칙

| 우선순위 | 포함 기준 | 제외 기준 |
|---------|-----------|----------|
| P1 (필수) | 사용자 요청에 직접 답하는 결론 | 팀 내부 운영 메모 |
| P1 (필수) | 숫자·수치·구체적 결과 | COLLAB/TEAM 태그 |
| P2 (권고) | 복수 팀이 언급한 이슈 (1회로 통합) | 중간 진행 상태 설명 |
| P2 (권고) | PM의 판단·권고 | "향후 검토 예정" 수준의 모호한 계획 |
| P3 (선택) | 단일 팀만 언급한 세부 사항 | 로컬 파일 경로 |
| P3 (선택) | 위험·이슈·충돌 | 팀별 역할 설명 |

**근거**: HBR 연구 — 인간 단기기억은 5개 개념 동시 처리 한계. P1만으로 핵심 전달, P2는 판단 지원, P3는 부록 처리.

### 2-2. 정제·요약 기준

| 항목 | 기준 | 근거 |
|------|------|------|
| 길이 | 본문 15줄 이내 (텔레그램 기준) | 스크롤 없이 읽을 수 있는 분량 |
| 결론 | 1문장, 50자 이내 | BLUF 원칙 — 첫 문장에 핵심 배치 |
| 핵심 발견 | 3~5개 항목, 항목당 1줄 | 피라미드 원칙 — 논거 3~5개 |
| 헤더 | 결론을 담은 명사구 (동사 포함 가능) | NN/g — 독자는 헤더만 스캔 |
| 사실 vs 의견 구분 | 사실은 수치와 함께, 의견은 "PM 판단:"으로 표기 | 신뢰성 확보 |

**Before/After 예시**:

❌ Before (비정제):
```
Engineering 팀 보고: API 구현이 완료됐습니다. 배포 과정에서 여러 가지
어려움이 있었으며 수동 작업이 필요했습니다.
Ops 팀 보고: 배포 파이프라인이 수동 상태이며 자동화가 필요합니다.
```

✅ After (정제):
```
## 배포 자동화 부재로 스프린트 완료 지연
- Engineering: API 구현 완료, 배포 수동 작업 **2~3시간** 소요
- Ops: 파이프라인 미자동화 확인 (Engineering과 동일 이슈 → 통합)
- **PM 권고**: 다음 스프린트 전 CI/CD 구축 우선 착수
```

### 2-3. 중복 제거 룰

**5단계 클러스터링 프로세스**:
```
1단계: 부서별 결과 수신 (원문 보존)
2단계: 주제별 클러스터링 (팀 이름 기준 X, 이슈/발견사항 기준 O)
3단계: 동일 주제 내 중복 제거 → 최신·고신뢰 출처 우선, 수치가 있으면 수치 우선
4단계: 빠진 관점 확인 (MECE: 누락 없는가)
5단계: 섹션별 1개 포인트 원칙으로 배치
```

**병합 기준**:
- 동일 이슈 2개 팀 언급 → 1줄로 통합, 출처 병기 생략 (팀 이름 나열 금지)
- 유사 문장 (70% 이상 의미 중복) → 더 구체적인 문장 채택
- 수치 충돌 → 최신 결과 우선, 충돌 사실을 "위험·이슈" 섹션에 명시

## 3. 표준 섹션 구조 (Phase 2 산출물)

### 3-1. 5섹션 표준 포맷

```
## 결론
[1문장: 무엇이 어떻게 됐는지 + PM 판단/권고. 50자 이내. 필수.]

## 핵심 발견사항
[MECE 원칙으로 3~5개 항목. 팀별 나열 금지. 주제별 통합. 필수.]

## 위험·이슈 (있을 때만)
[리스크, 충돌, 미완료 항목. 없으면 섹션 전체 생략.]

## PM 결정 및 권고
[PM의 명시적 판단 + 다음 단계 제안. 필수. 최소 1개.]

## 다음 조치 (있을 때만)
[실제 후속 태스크 목록. 없으면 섹션 전체 생략.]
```

### 3-2. 섹션별 작성 가이드라인

| 섹션 | 분량 | 포함 | 제외 |
|------|------|------|------|
| 결론 | 1문장 | 핵심 결과 + PM 권고 | 배경 설명, 과정 설명 |
| 핵심 발견사항 | 3~5 bullet | 사실·수치·완료 항목 | 팀 이름 기준 분류 |
| 위험·이슈 | 1~3 bullet | 미해결 문제, 충돌, 리스크 | 해결된 이슈 |
| PM 결정 및 권고 | 2~4 bullet | PM 판단, 우선순위 권고 | 팀 보고 요약 반복 |
| 다음 조치 | 1~5 bullet | 담당자·기한 포함 구체적 태스크 | 모호한 계획 |

### 3-3. 정보 흐름 (스토리라인)

```
[결론] → "그래서 뭐야?" (1문장으로 즉시 답)
    ↓
[핵심 발견사항] → "왜 그런 결론이야?" (3~5개 근거)
    ↓
[위험·이슈] → "걱정할 게 있어?" (있으면 명시)
    ↓
[PM 결정 및 권고] → "PM은 뭘 권하지?" (명시적 판단)
    ↓
[다음 조치] → "그럼 뭘 해야 해?" (구체적 실행)
```

### 3-4. GAP 매핑표 (현행 vs 개선안)

| 섹션 | 현행 | 개선안 | GAP 원인 |
|------|------|--------|---------|
| 결론 | `## 결론` (1문장, 존재) | `## 결론` (1문장 + PM 권고 포함) | PM 권고가 빠지거나 하단에 위치 |
| 근거 | `## 핵심 내용` (팀별 나열) | `## 핵심 발견사항` (주제별 통합) | MECE 클러스터링 없음 |
| 이슈 | `## 판단 근거` (중복, 불명확) | `## 위험·이슈` (리스크 전용) | "판단 근거"가 핵심내용과 겹침 |
| 권고 | 없음 (결론에 묻힘) | `## PM 결정 및 권고` (전용 섹션) | PM 역할 희석 |
| 후속 | `## 다음 조치` (있을 때만) | `## 다음 조치` (동일, 유지) | 유지 |

## 4. Before/After 샘플 (Phase 3 산출물)

### 사례: 멀티 부서 취합 후 최종보고

**Before (현행 — 바이패싱 형태)**:
```
## 결론
이번 작업이 완료되었습니다.

## 핵심 내용
Engineering 팀: API 구현을 완료했습니다. 배포 과정에서 수동 작업이
필요했고 2~3시간 소요됐습니다. CI/CD 파이프라인 구축이 필요합니다.

Ops 팀: 배포 파이프라인이 자동화되지 않아 수동으로 진행했습니다.
모니터링 대시보드 추가 구성이 필요합니다.

Design 팀: 와이어프레임을 완료했습니다. 개발팀 연계를 기다리고 있습니다.

## 다음 조치
각 팀에서 위 작업을 진행합니다.
```

**After (개선안 — PM 통합 관점)**:
```
## 배포 자동화 구축이 최우선 — CI/CD 미비로 스프린트 지연 발생
API·디자인은 완료됐으나, 배포 자동화 부재(Engineering·Ops 공통 확인)로
매 배포당 **3시간** 수동 작업이 발생 중. 다음 스프린트 전 CI/CD 착수 권고.

## 핵심 발견사항
- API 구현 완료, 기능 정상 동작 확인
- 와이어프레임 검수 완료, 개발 연계 대기
- 배포 수동 작업 **3시간/회** 발생 (Engineering·Ops 공통 이슈 → 통합)
- 모니터링 대시보드 미구성

## 위험·이슈
- CI/CD 미구축 → 릴리즈 속도 저하, 휴먼 에러 위험

## PM 결정 및 권고
- **P1**: 다음 스프린트 내 CI/CD 파이프라인 구축 착수
- **P2**: 모니터링 대시보드 구성 (CI/CD 완료 후)
- API·디자인은 다음 단계 진행 가능

## 다음 조치
- [ ] Ops: CI/CD 파이프라인 설계 및 구축 (이번 주)
- [ ] Engineering: 파이프라인 연동 테스트 (다음 주)
```

## 5. 작성자 자가점검 체크리스트 (Phase 3 산출물)

보고서 발송 전 아래 10개 항목을 순서대로 점검한다.

```
[ ] 1. 결론 섹션이 첫 번째이고 1문장인가?
[ ] 2. 결론에 PM 판단/권고가 포함되어 있는가?
[ ] 3. 핵심 발견사항이 팀별이 아닌 주제별로 분류됐는가?
[ ] 4. 동일 내용이 두 섹션 이상에 반복되지 않는가? (MECE)
[ ] 5. 수치·결과·사실은 구체적인가? (숫자 없는 추상 표현 최소화)
[ ] 6. "PM 결정 및 권고" 섹션이 있고 PM 의견이 명시됐는가?
[ ] 7. 팀 이름으로만 구분된 섹션이 없는가? (팀별 원문 bypass 금지)
[ ] 8. 내부 운영 태그(COLLAB/TEAM/ARTIFACT/로컬 경로)가 제거됐는가?
[ ] 9. 본문이 15줄 이내인가? (넘으면 부록으로 이동)
[ ] 10. "위험·이슈"와 "다음 조치"가 없을 때 해당 섹션이 생략됐는가?
```

**합격 기준**: 10개 중 8개 이상 통과. 미달 시 재작성.

## 6. 향후 개정 기준 (운영 지침)

| 조건 | 개정 트리거 | 담당 |
|------|------------|------|
| 사용자가 "보고서가 읽기 어렵다" 3회 이상 | 섹션 구조 재검토 | PM |
| 부서 추가/삭제 | 취합 원칙 P2/P3 업데이트 | PM |
| LLM 모델 교체 | 프롬프트 캘리브레이션 | Engineering |
| 보고서 평균 길이가 20줄 초과 지속 | 길이 기준 강화 | PM |
| "PM 권고 없음" 피드백 2회 이상 | PM 결정 섹션 프롬프트 강화 | PM |

**개정 절차**: 이 문서 버전 업 → `_SYNTHESIS_PROMPT` / `ensure_user_friendly_output` 프롬프트 동시 업데이트 → 커밋 → 봇 재기동
