## Phase 1: 실행 순서 타임라인 분석

### 계획 vs 실제 실행 순서

```
[PM 의도]         리서치(T-238) → 개발실(T-239) → 운영실(T-240)
                       ↓               ↓               ↓
[실제 실행]  T-238 생성 ──→ T-239 생성(LLM, ~18s 후) 직후 TaskPoller 선점
                                 ↓
                        T-238, T-239, T-240 병렬 실행 시작
```

### 타임라인 (추정, TaskPoller 2초 주기 기준)

| 시각 (t=0 기준) | 이벤트 |
|----------------|--------|
| t+0s | PM `_dispatch_subtasks` 시작, T-238(리서치) 생성 + 발송 |
| t+0s~18s | LLM이 T-239(개발실) 프롬프트 생성 중 |
| t+18s | T-239 DB 등록 (pending) — 아직 deps 미등록 상태 |
| t+18~20s | TaskPoller가 T-239를 "deps 없음 = 즉시 실행" 으로 판단 → 개발실 선점 |
| t+36s | T-240(운영실) 생성 후 동일 패턴으로 선점 |
| 결과 | T-238, T-239, T-240 사실상 병렬 실행 |

### 계획 대비 실제 불일치 목록

| 항목 | 계획 | 실제 |
|------|------|------|
| 실행 순서 | 순차 (리서치→개발→운영) | 병렬 (3개 동시) |
| 의존성 등록 시점 | 태스크 생성 직후 | 태스크 생성 루프 완료 후 (지연) |
| 개발실 시작 조건 | T-238 완료 후 | T-238 완료 무관하게 즉시 시작 |

## Phase 2: 원인 분석

### 원인 분류 분석표

#### 🔴 구조적 원인 (코드 설계 결함)

**레이스 컨디션 — 의존성 등록 전 TaskPoller 조기 수신**

```
[수정 전 흐름]
Step 1: 태스크 생성 루프 (LLM 호출, ~15-20s/태스크)
          └─ 각 태스크 DB 등록 완료
Step 2: 의존성 등록 (for i, st in enumerate(subtasks))
Step 3: 첫 웨이브 발송

문제: Step 1 에서 T-239가 DB에 생성되는 순간,
      TaskPoller(2초 주기)가 deps 없는 태스크로 인식 → 즉시 수신
      Step 2(deps 등록)는 모든 태스크 생성 완료 후에야 실행되므로
      항상 Step 2 전에 Poller가 선점할 수 있었음.
```

```
[수정 후 흐름] (commit: cf42da4)
Step 0: 태스크 ID 사전 생성 (LLM 없음, 빠름)
        + 모든 deps를 pm_task_dependencies에 먼저 등록
Step 1: 태스크 생성 루프 (LLM 호출)
Step 2: deps 등록 — Step 0에서 완료됨 (skip)
Step 3: 첫 웨이브 발송

보장: T-239가 DB에 생성될 때 이미 T-239→T-238 dep이 등록되어 있음
     → TaskPoller가 T-238 완료 전 T-239를 수신하지 않음
```

#### 🟡 운영적 원인

- PM이 순차 실행 의도였지만 배분 메시지에 **DEPENDS 선언이 명시적으로 포함되지 않은** 경우도 있었음
- PM 프롬프트에 "리서치 먼저, 그 다음 개발실" 같은 자연어 순서 표현이 있었으나, 시스템 레벨 의존성으로 자동 변환하는 로직이 LLM 판단에 의존

#### 🟠 시스템적 원인

- TaskPoller 폴링 주기(2초)가 LLM 태스크 생성 속도(~18초/태스크)보다 훨씬 빠름
- `pm_task_dependencies` 테이블 조회 시 "등록된 deps 없음 = 즉시 실행 가능"으로 판단하는 로직이 "아직 등록 전"과 "진짜 deps 없음"을 구분하지 못했음

### 영향도 평가 매트릭스

| 원인 | 재발 가능성 | 영향도 | 수정 여부 |
|------|------------|--------|-----------|
| deps 등록 순서 역전 (구조적) | 수정 전: 100% / 수정 후: 0% | 높음 (순서 보장 완전 붕괴) | ✅ cf42da4 |
| PM 배분 메시지 DEPENDS 미명시 (운영적) | 중간 (LLM 판단 의존) | 중간 (순서 힌트 유실) | 🟡 SKILL.md 가이드 추가 |
| Poller vs LLM 속도 불균형 (시스템적) | 수정 전: 높음 / 수정 후: 해소 | 높음 | ✅ cf42da4로 해소 |
