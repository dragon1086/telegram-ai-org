## Phase 1: 버그별 재현 시나리오

### BUG-01: 한국어 조사 미분리

**발생 조건**
- LLM이 `[COLLAB:...]` 태그에서 조사가 붙은 한국어 부서명/도메인 용어를 사용할 때
- `re.split(r"\W+", task.lower())` 호출 시 한글은 `\w` (Unicode word char)로 분류되어 분리 안 됨

**실제 출력 vs 기대 출력**

| MRE 입력 (task 문자열) | 실제 words 집합 | 기대 words 집합 |
|----------------------|----------------|----------------|
| "개발실에 API 구현 요청" | `{"개발실에", "api", "구현", "요청"}` | `{"개발실에", "개발실", "api", "구현", "요청"}` |
| "디자인팀을 통해 UI 리뷰" | `{"디자인팀을", "통해", "ui", "리뷰"}` | `{"디자인팀을", "디자인팀", ...}` |
| "운영실로 배포 요청" | `{"운영실로", "배포", "요청"}` | `{"운영실로", "운영실", ...}` |
| "마케팅팀이 필요한 카피" | `{"마케팅팀이", "필요한", "카피"}` | `{"마케팅팀이", "마케팅팀", ...}` |
| "기획실의 PRD 리뷰 요청" | `{"기획실의", "prd", "리뷰", "요청"}` | `{"기획실의", "기획실", ...}` |

**주요 조사 유형별 MRE**

| 조사 | 예시 입력 | 실제(오류) | 기대(정상) |
|------|----------|-----------|-----------|
| `을/를` | "개발실을 통해" | "개발실을" 매치 실패 | "개발실" 매치 성공 |
| `이/가` | "디자인팀이 담당" | "디자인팀이" 매치 실패 | "디자인팀" 매치 성공 |
| `은/는` | "운영실은 배포 담당" | "운영실은" 매치 실패 | "운영실" 매치 성공 |
| `으로/로` | "운영실로 요청" | "운영실로" 매치 실패 | "운영실" 매치 성공 |
| `에서` | "개발실에서 구현" | "개발실에서" 매치 실패 | "개발실" 매치 성공 |
| `의` | "기획실의 PRD" | "기획실의" 매치 실패 | "기획실" 매치 성공 |

## Phase 2: 근본 원인 심층 분석 (Why-5)

### BUG-01 Why-5

```
Why 1: target_org = None 반환 → COLLAB 태그가 chat fallback으로만 처리됨
Why 2: score가 confidence threshold(≥2)를 충족 못함
Why 3: words 집합에 "개발실에"가 있고 haystack에는 "개발실"만 있어 매치 실패
Why 4: re.split(r"\W+")가 한글 조사를 분리하지 못함
Why 5: Python의 \W는 Unicode word chars(한글 포함)를 단어 문자로 취급 — 언어별 형태소 분석 없음
```

**계층 분류**

| 계층 | 원인 포함 여부 | 설명 |
|------|--------------|------|
| 입력처리 | ✅ **주원인** | `_tokenize` 단계에서 언어 구분 없이 `\W+` 분리 |
| 비즈니스로직 | ○ 부분 | threshold `score >= 2` 조건이 oops를 증폭 |
| 출력처리 | ✗ | 무관 |

**한국어 처리 특화 원인**
- Python `re` 모듈의 `\W`는 `[^a-zA-Z0-9_]` 대신 Unicode property `\P{Word}`를 따름
- 한글 자모(`\uAC00-\uD7A3`)는 Word character로 분류됨
- 따라서 "개발실에"는 하나의 word token으로 인식

## Phase 3: 수정 방안 설계 및 트레이드오프

### BUG-01 수정 방안 비교

| 방안 | 접근법 | 수정 범위 | 부작용/회귀 위험 | 난이도 |
|-----|--------|----------|----------------|--------|
| **(A) 규칙 기반 후처리** ✅ **권장** | 주요 조사 목록(`을/를/이/가/은/는/으로/로...`)으로 어간 추출 후 원형+어간 모두 words에 추가 | 최소 (2개 메서드 내부만) | 조사 목록 외 접미사 처리 안 됨, 영어 오탐 없음 | **L** |
| (B) 형태소 분석기 도입 (konlpy/mecab) | 외부 라이브러리로 형태소 분리 | 전면 (의존성 추가, 인프라 변경) | konlpy 설치 실패 시 배포 차단, 느린 초기화 | **H** |
| (C) LLM 후처리 | LLM API 재호출로 단어 추출 | 전면 (비용 증가, 레이턴시 +300ms) | API 비용/레이턴시 증가, 실패 시 전체 COLLAB 드롭 | **H** |

**권장 방안 (A) 선택 근거**:
- 의존성 추가 없이 즉시 적용 가능
- 주요 조사(~20종) 목록으로 실용적 커버리지 확보
- 원형+조사 형태 모두 words에 추가하므로 기존 영어/숫자 처리 영향 없음
- 형태소 분석기 대비 초기화 비용 0ms

**구현**: `_tokenize_task()` static method 추가 → `_infer_collab_target_org()`, `_infer_collab_target_mentions()` 양쪽 적용 완료.
