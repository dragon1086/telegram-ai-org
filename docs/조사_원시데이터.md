## 1. PyPI 패키징 표준

### 1-A. PEP 요약

| PEP | 제목 | 핵심 내용 | 출처 |
|-----|------|-----------|------|
| **PEP 517** (2017) | Build-system independent format | 빌드 백엔드 인터페이스 표준화. `build_wheel()`, `build_sdist()` 등 후크 정의. setup.py 직접 실행 탈피. | [peps.python.org/pep-0517](https://peps.python.org/pep-0517/) |
| **PEP 518** (2016) | Minimum build system requirements | `pyproject.toml` 파일 도입. `[build-system].requires` 로 빌드 의존성 선언. pip가 빌드 전 격리 환경 구성 가능해짐. | [peps.python.org/pep-0518](https://peps.python.org/pep-0518/) |
| **PEP 621** (2021) | Storing project metadata in pyproject.toml | `[project]` 테이블 표준화. name/version/dependencies/scripts 등 메타데이터를 백엔드 중립적으로 선언. setup.cfg 대체 목표. | [packaging.python.org](https://packaging.python.org/en/latest/guides/writing-pyproject-toml/) |
| **PEP 639** (2024) | Improving License Clarity | SPDX 표현식으로 `license = "MIT"` 직접 사용. License classifier deprecation. | [peps.python.org/pep-0639](https://peps.python.org/pep-0639/) |

- **적용 가능성**: 현 프로젝트 `pyproject.toml`은 이미 PEP 517/518/621 준수. PEP 639 (SPDX `license = "MIT"`) 도 적용됨. ✅

### 1-B. 패키징 방식 비교 원시 데이터

| 항목 | pyproject.toml | setup.cfg | setup.py |
|------|---------------|-----------|----------|
| **도입 시기** | PEP 518 (2016), PEP 621 (2021) | setuptools 30.3.0 (2016) | Python 초기부터 |
| **포맷** | TOML (정적 선언) | INI (정적 선언) | Python 코드 (동적) |
| **빌드 백엔드** | 모든 백엔드 (setuptools, flit, hatchling, poetry-core, uv-build) | setuptools 전용 | setuptools 전용 |
| **현재 권장** | ✅ 공식 표준 | 레거시 지원용 | 하위 호환성 유지 |
| **채택률** (2025 기준) | 신규 패키지 대부분, setuptools 사용자의 30% | setuptools 기존 프로젝트 | setuptools 미선언 패키지 ~70% (암묵적) |
| **동적 메타데이터** | `dynamic = ["version"]` 선언 후 백엔드 처리 | 미지원 (직접 코딩 필요) | 완전 지원 (코드 실행) |
| **보안** | 정적 파일 — 코드 실행 없음 | 정적 파일 — 코드 실행 없음 | pip install 시 임의 코드 실행 위험 |
| **도구 호환** | ruff, mypy, pytest, black 설정 통합 가능 | 일부 도구 지원 | 별도 config 파일 필요 |

- **출처**: [setuptools.pypa.io](https://setuptools.pypa.io/en/latest/userguide/pyproject_config.html), [labs.quansight.org/pep-517-build-system-popularity](https://labs.quansight.org/blog/pep-517-build-system-popularity), [mcginniscommawill.com/pyproject-toml](https://mcginniscommawill.com/posts/2025-01-26-pyproject-toml-explained/)

## 2. Docker 오픈소스 배포 베스트 프랙티스

### 2-A. Multi-Stage Build 패턴

| 단계 | 역할 | 베이스 이미지 | 핵심 포인트 |
|------|------|-------------|------------|
| **builder** | 의존성 설치 + wheel 빌드 | `python:3.11-slim` | build-essential 포함, pip install -e . |
| **node-installer** | Node CLI 선택 설치 | `node:20-slim` | ARG ENGINE으로 분기 |
| **runtime** | 최종 실행 이미지 | `python:3.11-slim` | COPY --from=builder만 사용 |

- **출처**: [docs.docker.com/build/building/multi-stage](https://docs.docker.com/build/building/multi-stage/), [docs.docker.com/build/building/best-practices](https://docs.docker.com/build/building/best-practices/)
- **적용 가능성**: 현 `Dockerfile`이 이미 3-stage 구조 채택. ✅

---

### 2-B. .dockerignore 표준 패턴

| 제외 대상 | 이유 |
|-----------|------|
| `.git`, `.github` | VCS 메타데이터 — 불필요 |
| `.venv`, `venv` | 가상환경 — 이미지 내 별도 설치 |
| `*.md` | 문서 파일 — 런타임 불필요 |
| `.env*` | 시크릿 — 절대 포함 금지 |
| `tests/`, `*.log` | 테스트/로그 — 프로덕션 불필요 |
| `dist/`, `build/` | 빌드 아티팩트 — 내부에서 재생성 |
| `__pycache__`, `*.pyc` | 캐시 — 오염 방지 |
| `.worktrees/` | (프로젝트 특화) worktree 디렉토리 |

- **출처**: [docs.docker.com/build/building/best-practices](https://docs.docker.com/build/building/best-practices/), [betterstack.com/docker-build-best-practices](https://betterstack.com/community/guides/scaling-docker/docker-build-best-practices/)

### 2-C. ENTRYPOINT vs CMD 패턴

| 패턴 | 코드 예시 | 사용 시나리오 |
|------|-----------|-------------|
| **ENTRYPOINT 고정 + CMD 기본인수** | `ENTRYPOINT ["python", "-m", "myapp"]` `CMD ["--help"]` | 단일 목적 이미지 |
| **쉘 스크립트 ENTRYPOINT** | `ENTRYPOINT ["/app/docker-entrypoint.sh"]` | 초기화 로직(env 검증, DB 마이그레이션) 필요 시 |
| **CMD만** | `CMD ["python", "main.py"]` | docker run 시 완전 교체 가능 원할 때 |

- **현 프로젝트**: `ENTRYPOINT ["python", "-m", "telegram_ai_org"]` + `CMD []` → 권장 패턴 준수 ✅

### 2-D. DockerHub / GHCR 배포 전략

| 레지스트리 | 특징 | 추천 사용 사례 |
|-----------|------|--------------|
| **DockerHub** | 가장 넓은 인지도, 무료 티어(pull 제한 있음) | 일반 사용자 대상 배포 |
| **GHCR** (ghcr.io) | GitHub Actions 네이티브 통합, 무제한 public | 오픈소스 프로젝트 권장 |
| **멀티 레지스트리** | 두 곳 동시 푸시 | 가용성 최대화 |

**GitHub Actions GHCR 배포 워크플로우 핵심 패턴**:
```yaml
on:
  push:
    tags: ['v*']
jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.ref_name }}
```

- **출처**: [packaging.python.org/github-actions](https://packaging.python.org/en/latest/guides/publishing-package-distribution-releases-using-github-actions-ci-cd-workflows/), [github.com/pypa/gh-action-pypi-publish](https://github.com/pypa/gh-action-pypi-publish)

## 3. 3엔진 원클릭 설치 레퍼런스 사례

### 3-A. claude-code (Anthropic Claude Code)

| 설치 방법 | 명령어 | 비고 |
|-----------|--------|------|
| **공식 Native 설치 (권장)** | `curl -fsSL https://claude.ai/install.sh \| bash` | Node.js 불필요, 자동 업데이트 |
| **Homebrew** | `brew install --cask claude-code` | macOS 전용, 수동 업데이트 필요 |
| **npm** | `npm install -g @anthropic-ai/claude-code` | Deprecated (native 권장) |
| **Windows PowerShell** | `irm https://claude.ai/install.ps1 \| iex` | Windows 전용 |

- **패키지 구조**: npm 패키지 `@anthropic-ai/claude-code`, Homebrew Cask `claude-code`
- **Homebrew Formula**: [formulae.brew.sh/cask/claude-code](https://formulae.brew.sh/cask/claude-code)
- **출처**: [code.claude.com/docs/en/setup](https://code.claude.com/docs/en/setup), [morphllm.com/install-claude-code](https://www.morphllm.com/install-claude-code)
- **적용 가능성**: `scripts/setup.sh`가 `which claude` 자동 감지. Docker Stage 2에서 npm 설치. ✅

### 3-B. codex (OpenAI Codex CLI)

| 설치 방법 | 명령어 | 비고 |
|-----------|--------|------|
| **npm** | `npm install -g @openai/codex` | 주 배포 채널 |
| **Homebrew** | `brew install --cask codex` | macOS/Linux |
| **바이너리 직접 다운로드** | GitHub Releases (tar.gz) | aarch64/x86_64, macOS/Linux/Windows |

- **패키지 구조**: npm `@openai/codex`, Homebrew Cask, Rust 바이너리 (Bazel + pnpm 빌드)
- **인증**: ChatGPT OAuth 또는 API Key
- **플랫폼**: macOS/Linux (안정), Windows (실험적)
- **출처**: [npmjs.com/@openai/codex](https://www.npmjs.com/package/@openai/codex), [github.com/openai/codex](https://github.com/openai/codex)
- **적용 가능성**: Docker Stage 2에서 `npm install -g @openai/codex` 설치 구현됨. ✅

### 3-C. gemini-cli (Google Gemini CLI)

| 설치 방법 | 명령어 | 비고 |
|-----------|--------|------|
| **npm (권장)** | `npm install -g @google/gemini-cli` | Node.js 20+ 필요 |
| **npx (임시 실행)** | `npx @google/gemini-cli` | 설치 없이 바로 실행 |
| **Homebrew** | `brew install gemini-cli` | macOS/Linux |

- **현재 버전**: v0.32.1 (2026-03-25 기준), Gemini 2.5 Flash 기반
- **인증**: Google OAuth 무료 (60 req/min, 1,000 req/day) 또는 API Key
- **패키지**: npm `@google/gemini-cli`, Homebrew Formula `gemini-cli`
- **Homebrew Formula**: [formulae.brew.sh/formula/gemini-cli](https://formulae.brew.sh/formula/gemini-cli)
- **출처**: [npmjs.com/@google/gemini-cli](https://www.npmjs.com/package/@google/gemini-cli), [github.com/google-gemini/gemini-cli](https://github.com/google-gemini/gemini-cli)
- **적용 가능성**: Docker Stage 2에서 `npm install -g @google/gemini-cli` 설치 구현됨. ✅

### 3-D. 엔진별 자동 감지 패턴 비교

| 엔진 | 바이너리명 | 자동 감지 명령 | 버전 확인 |
|------|-----------|--------------|----------|
| claude-code | `claude` | `which claude` | `claude --version` |
| codex | `codex` | `which codex` | `codex --version` |
| gemini-cli | `gemini` | `which gemini` | `gemini --version` |

- **스크립트 패턴** (현 `scripts/setup.sh` 기반):
  ```bash
  for engine in claude codex gemini; do
    if command -v "$engine" &>/dev/null; then
      AI_ENGINE="$engine"
      break
    fi
  done
  ```
