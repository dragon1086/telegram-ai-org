# Phase 3: 에이전트 실시간 모니터링 운영 가이드 (PM/개발팀용)

작성일: 2026-03-23
작성팀: 운영실 (aiorg_ops_bot)
배포 대상: PM, 개발팀, 운영팀

---

## 1. 이 가이드의 목적

에이전트가 **"일하다 잘린 것"인지 "stuck인지"** 를 Telegram에서 바로 판단할 수 있도록 안내한다.

이전에는 타임아웃이 오면 `⏰ 무응답 183초 (한도 180s)` 메시지만 보여 아무것도 알 수 없었다.
지금은 3가지 케이스로 자동 분류되어 어떤 행동을 취해야 할지 바로 알 수 있다.

---

## 2. 실시간 모니터링 방법

### 2-1. Telegram에서 직접 확인 (가장 쉬움)

에이전트가 작업 중이면 **30초마다 자동으로** Telegram 메시지가 업데이트된다.

**정상 작동 중 표시:**
```
⏳ 처리 중... (3.2분 경과, heartbeat #6)
마지막: orchestration.yaml 파일 읽는 중
```
→ 이 메시지가 보이면 에이전트가 살아서 일하고 있는 것이다. 기다리면 된다.

**주의 징후:**
```
⏳ 처리 중... (8.0분 경과, heartbeat #1)
```
→ heartbeat가 1회인데 8분이 지났다면 LLM 응답이 매우 느린 상태.

### 2-2. 타임아웃 메시지 해석 (3케이스)

타임아웃(300초) 발생 시 아래 3가지 중 하나가 표시된다.

| 메시지 패턴 | 의미 | PM 대응 행동 |
|---|---|---|
| `마지막 출력 Xs 전: <내용> [heartbeat N회 — 작업 중 잘렸을 가능성]` | 에이전트가 열심히 작업하다 타임아웃으로 종료됨 | **재시도**. 단순히 시간이 더 필요했던 것. 반복 시 타임아웃 증가 요청 |
| `실행 중 출력 없음 [heartbeat N회 — LLM 응답 대기 중]` | Claude API 응답 지연 중. 프로세스는 살아있었음 | **잠시 후 재시도** (5~10분). Claude API 상태 이상이면 계속 실패할 수 있음 |
| `실행 중 출력 없음 [heartbeat 0회 — 작업 시작 전 stuck]` | 태스크가 시작조차 되지 않았음. 진짜 stuck | 태스크 **재배정** 또는 봇 상태 점검 (health_check.py 실행) |

### 2-3. health_check.py 실행 (봇 전체 상태)

서버에서 직접 확인 가능:
```bash
cd /Users/rocky/telegram-ai-org
python scripts/health_check.py
```

**정상 출력:**
```
aiorg_pm_bot          UP  84974
...
✅ 모든 봇 정상 동작 중
```

**이상 출력:**
```
aiorg_pm_bot          DOWN -
⚠️ 일부 봇이 다운 상태입니다
```
→ DOWN인 봇은 `bash scripts/request_restart.sh --reason "봇 다운 감지"` 로 재기동 요청

### 2-4. 로그 실시간 확인

```bash
# heartbeat + 타임아웃만 필터링
tail -f /Users/rocky/telegram-ai-org/logs/claude_sessions.log \
  | grep -E "heartbeat|무응답|timeout|stuck"

# agent_monitor 이벤트 (stuck 감지 + 자동 주입 기록)
tail -f ~/.ai-org/agent-monitor.log
```

## 3. 상태 지표 해석 기준

| 지표 | 정상 | 주의 | 비정상 |
|---|---|---|---|
| Telegram 진행 메시지 업데이트 | 30초 이내 | 60초 이상 미업데이트 | 5분 이상 변화 없음 |
| heartbeat 횟수 | 경과시간/30 ≈ hb 횟수 | heartbeat 1~2회인데 2분 경과 | heartbeat 0회인데 타임아웃 |
| 타임아웃 케이스 | 없음 | "작업 중 잘림" 발생 | "작업 시작 전 stuck" 발생 |
| 봇 프로세스 상태 | 전원 UP | 1개 DOWN | 2개 이상 DOWN |

## 4. FAQ

**Q: 타임아웃 메시지가 왔는데 재시도해야 하나요?**
> A: 케이스를 먼저 확인하세요.
> - `heartbeat N회 — 작업 중 잘렸을 가능성` → 재시도하세요. 작업이 길었을 뿐입니다.
> - `heartbeat 0회 — 작업 시작 전 stuck` → 봇 상태를 먼저 점검하세요.

**Q: stuck인지 작업 중인지 구분하는 가장 빠른 방법은?**
> A: 타임아웃 메시지의 `heartbeat N회` 숫자를 보세요.
> - N=0: 시작도 못 한 stuck
> - N>0 + 출력 있음: 작업하다 잘림 → 재시도
> - N>0 + 출력 없음: LLM 응답 대기 → 잠시 후 재시도

**Q: 타임아웃이 계속 반복된다면?**
> A: 다음 순서로 확인하세요.
> 1. Claude API 상태 확인 (LLM 응답 대기 케이스라면)
> 2. 태스크 자체가 너무 복잡한지 검토 (작업 중 잘림 케이스라면)
> 3. 운영팀에 `BOT_IDLE_TIMEOUT_SEC` 증가 요청 (현재 300s → 예: 600s)

**Q: 타임아웃 한도를 더 늘릴 수 있나요?**
> A: .env의 `BOT_IDLE_TIMEOUT_SEC` 값을 변경하면 됩니다. 변경 후 재기동 요청 필요.
> 운영팀에 요청하세요.

**Q: 작업 중인 에이전트를 강제로 중단하고 싶다면?**
> A: Telegram에서 해당 봇에 새 메시지를 보내면 기존 작업이 종료됩니다.
> 봇 프로세스를 직접 종료하지 마세요 (결과 유실 위험).
