# 오버스콥 방지 운영 가이드

> 작성일: 2026-03-23 | 버전: 1.0
> 적용 범위: 전체 조직 (PM + 모든 specialist)

---

## 1. 오버스콥이란?

**오버스콥(Over-scope)**: 에이전트가 PM이 지시한 범위를 초과하여 추가 구현·파일 생성·시스템 변경을 수행하는 행위.

### 실제 발생 패턴 (인시던트 기반)

| 패턴 | 예시 | 발생 조직 |
|------|------|----------|
| 분석 요청 → 구현 수행 | 리서치 요청에 분석 파일 + 자동화 스크립트 생성 | 개발실, 기획실 |
| 타 조직 영역 침범 | 개발실이 기획 PRD를 작성하거나 인프라 설정 변경 | 개발실 |
| 자기 개선 자율 실행 | 태스크 완료 후 "이왕이면" 리팩터링, 스킬 업데이트 | 전체 |
| 분석 완료 후 무단 구현 | "분석 결과에 따라 바로 구현했습니다" | 개발실, 성장실 |
| 불필요한 파일 생성 | 보고만 하면 될 것을 마크다운 파일 10개 생성 | 전체 |

## 2. 오버스콥 자가 진단 체크리스트 (착수 전)

태스크를 수신하면 실행 전 아래 3가지를 확인한다.

```
[ ] 1. 태스크 유형 확인
    - "분석/조사/검토/리서치/비교" → 보고서 산출만 (코드/파일 생성 금지)
    - "구현/만들어라/개발/작성하라" → 코드/파일 생성 허용
    - 키워드 없음 → PM에게 "구현 포함인지?" 확인 후 착수

[ ] 2. 산출물 범위 확인
    - 배분 메시지의 "기대 산출물"에 명시된 것만 생성
    - 명시 없는 파일/코드 생성 = 오버스콥

[ ] 3. 도메인 확인
    - 내 전문 영역(role) 밖의 작업이면 PM에게 [COLLAB] 위임 요청
    - 내 영역이지만 PM이 지시하지 않은 작업 = 오버스콥
```

## 3. PM이 태스크 배분 시 필수 포함 항목

오버스콥의 절반 이상은 PM이 범위를 모호하게 배분하기 때문에 발생한다.

### 배분 메시지 필수 포맷

```
[태스크 배분] #{태스크ID}
담당: @{봇이름}
내용: {태스크 설명}
긴급도: 즉시 / 오늘중 / 이번주
실행 범위 유형: [분석/보고서] | [구현/파일생성] | [혼합]
기대 산출물: {구체적 파일명 또는 형식}
실행 범위: {이 봇이 해야 할 것만 명시 — 범위 외 작업은 별도 서브태스크}
협업 필요: {다른 봇 있으면 명시 / 없으면 "없음"}
```

> **"실행 범위 유형"이 없으면 봇은 PM에게 확인을 요청해야 한다.**
> PM이 "분석/보고서"라고 명시했으면 봇은 절대 구현하지 않는다.

## 4. 조직간 불일치 감지 및 조율 워크플로우

### 시스템 자동 처리 (코드 레벨)

`core/result_synthesizer.py`의 `ResultSynthesizer`가 판정:
- `SUFFICIENT` → 통합 보고서 작성 진행
- `NEEDS_INTEGRATION` → PM이 추가 작업 요청
- `CONFLICTING` → **자동 조율 태스크 dispatch** (수동 개입 불필요)

`CONFLICTING` 발동 조건:
- 같은 항목(라이선스, 수치, 결론)에 대해 조직간 상반된 결과
- LLM이 FOLLOW_UP을 제공하면 그것을 조율 태스크로 변환
- FOLLOW_UP 없으면 product_bot → research_bot 순서로 fallback 자동 생성

### PM이 직접 감지해야 하는 경우 (자동 탐지 미발동 시)

ResultSynthesizer가 탐지 못한 미묘한 불일치:

| 발동 기준 | 예시 |
|----------|------|
| 동일 사실에 대한 상반 결론 | 기획실 "Apache 2.0" vs 개발실 "MIT" |
| 상반된 기술 권고 | 리서치 "LangGraph 도입" vs 기획 "직접 개발" |
| PM이 결론 판단 불가 | 법률적 해석, 기술적 타당성 판단 |

이 경우 `pm-discussion` 스킬 수동 발동:
```
/pm-discussion "불일치 해소 필요: {항목}"
조직A 결론: {요약}
조직B 결론: {요약}
→ 근거와 출처를 제시하라
```

## 5. 에이전트 추가/수정 시 오버스콥 방지 체크리스트

새 조직 추가 또는 기존 조직 수정 시:

```
[ ] organizations.yaml preferred_skills에 task-kickoff 포함 확인
[ ] bot yaml instruction에 role 범위 명시 확인
[ ] orchestration.yaml global_instructions 적용 확인 (자동 주입)
[ ] 분석/구현 구분 테스트: "X를 분석해달라" 요청 시 파일 생성 없이 보고만 하는지 확인
[ ] 도메인 경계 테스트: 타 조직 영역 요청 시 PM 재위임 응답하는지 확인
```

## 6. 검증 시나리오 및 예상 응답

### 시나리오 A: 분석만 요청

**입력**: "TradingAgents 라이선스를 조사해달라"
**올바른 응답**: 라이선스 정보를 텍스트로 보고 (파일 생성 없음)
**오버스콥 응답**: `license_analysis.md` 파일 생성 + 비교표 + 추가 조사 파일 생성

### 시나리오 B: 범위 외 요청

**입력**: 개발실에게 "마케팅 전략도 함께 수립해달라" 요청
**올바른 응답**: "마케팅 전략은 성장실 영역입니다. PM에게 성장실 배분을 요청합니다."
**오버스콥 응답**: 개발실이 직접 마케팅 전략 문서 작성

### 시나리오 C: 불일치 발견

**상황**: 기획실 "Apache 2.0", 개발실 "MIT" 판정
**올바른 응답**: ResultSynthesizer CONFLICTING → 자동 조율 태스크 생성 OR PM이 `/pm-discussion` 발동
**오버스콥 응답**: "두 조직의 결과가 다릅니다. A는 Apache, B는 MIT라고 합니다." (불일치를 그냥 나열)

## 7. 관련 파일 위치

| 파일 | 역할 |
|------|------|
| `orchestration.yaml` → `global_instructions` | 전체 조직에 자동 주입되는 스코프 원칙 |
| `CLAUDE.md` → PM 업무 스코프 준수 원칙 | 분석/조사 요청 시 구현 금지 체크리스트 |
| `skills/task-kickoff/SKILL.md` | 태스크 착수 전 스코프 확인 5단계 |
| `skills/pm-task-dispatch/SKILL.md` | PM 배분 시 범위 유형 명시 + 불일치 감지 |
| `skills/pm-discussion/SKILL.md` | 조직간 불일치 조율 (자동 발동 조건 포함) |
| `core/result_synthesizer.py` | CONFLICTING 자동 감지 엔진 |
| `core/pm_orchestrator.py` | CONFLICTING 분기 처리 및 조율 태스크 dispatch |
