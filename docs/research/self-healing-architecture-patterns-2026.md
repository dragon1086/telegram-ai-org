## Phase 1: 핵심 용어 정의 및 조사 범위

### 핵심 용어 정의

| 용어 | 정의 |
|------|------|
| **Self-Healing System** | 장애 또는 품질 저하를 자동으로 감지하고 원인을 진단하여 사람의 개입 없이 복구/개선하는 시스템. "탐지 → 진단 → 실행 → 검증"의 폐쇄 루프(closed-loop)로 작동 |
| **Code Health Report** | 정적 분석, 린팅, 테스트 커버리지, 보안 스캔 등의 결과를 종합한 코드 품질 상태 보고서. 개선 트리거의 원천 데이터 |
| **Automated Remediation** | 코드 건강 리포트의 이슈를 사람이 수동으로 수정하는 대신, 에이전트/스크립트가 자동으로 수정 작업을 수행하는 과정 |
| **Feedback Loop** | 자동 개선 결과가 다음 주기의 리포트에 반영되어 시스템이 지속적으로 진화하는 순환 구조 |
| **Continuous Improvement Pipeline** | 코드 품질 측정 → 임계값 판단 → 자동 수정 → 검증 → 재측정의 단계가 반복되는 CI/CD 파이프라인 확장 구조 |
| **Observability-Driven Automation** | 메트릭/로그/트레이스 등 관측 데이터를 기반으로 자동화 액션을 트리거하는 방식. 임계값 초과 시 자동으로 Runbook 실행 |
| **Policy-as-Code** | 자동화 허용 범위, 승인 조건 등을 코드(YAML/OPA)로 선언하여 에이전트 행동을 통제하는 거버넌스 계층 |
| **Quality Gate** | CI 파이프라인 내 코드 품질 기준점. 미달 시 머지 차단. SonarQube의 핵심 개념 |

### 조사 키워드 목록
- `self-healing system`, `automated remediation pipeline`
- `code quality feedback loop`, `continuous improvement pipeline`
- `observability-driven automation`, `agentic SRE`
- `code health report trigger`, `AI agent code fix loop`
- `quality gate automated PR`, `Renovate Dependabot automerge`

### 조사 범위
- **언어/플랫폼**: Python, TypeScript, 언어 무관 CI/CD 레이어
- **도구 유형**: 정적 분석 → 자동 수정 → PR 생성 → 승인 → 머지
- **시점**: 2024~2026-03 기준 (최신 GA 도구 중심)

### 3-1. 트리거 방식 분류

| 유형 | 설명 | 예시 |
|------|------|------|
| **임계값 기반** | 메트릭이 기준값 초과 시 발동 | 커버리지 < 70%, 크리티컬 이슈 > 0 |
| **스케줄 기반** | 정기적으로 리포트 생성 후 액션 | Renovate 주 1회 의존성 스캔 |
| **이벤트 기반** | CI 실패, PR 생성, 커밋 푸시 등 | CI lint 실패 → LLM 에이전트 즉시 호출 |
| **예측 기반** | 과거 패턴 학습 후 문제 발생 전 선제 대응 | 성능 저하 패턴 인식 → 사전 스케일링 |

### 3-3. 자동화 수준 분류

| 수준 | 구조 | 장점 | 단점 | 적합 범주 |
|------|------|------|------|----------|
| **L1: 알림만** | 리포트 → 알림 | 안전, 구현 쉬움 | 효과 없음 | - |
| **L2: PR 제안** | 리포트 → 에이전트 → PR 생성 | 안전, 효과적 | 인간 검토 필요 | 코드 스멜, 보안 수정 |
| **L3: 조건부 자동 머지** | PR 생성 + CI 통과 → 자동 머지 | 빠름, 반복 작업 자동화 | 거버넌스 설정 필요 | 의존성 업데이트, 포맷팅 |
| **L4: 완전 자율** | 감지→수정→배포 전 과정 자동 | 최고 속도 | AI 취약점 주입 위험 (37.6%↑) | 위험 낮은 단순 작업만 |

### 3-4. 개선 대상 범주별 자동화 적합성

| 개선 대상 | 자동화 적합성 | 권장 수준 | 비고 |
|----------|------------|----------|------|
| **의존성 업데이트** | ★★★★★ 매우 높음 | L3 (조건부 자동 머지) | Renovate/Dependabot + CI 통과 조건 |
| **코드 포맷팅/린팅** | ★★★★☆ 높음 | L3 (자동 커밋) | ruff --fix, black 자동 적용 |
| **테스트 커버리지** | ★★★☆☆ 중간 | L2 (PR 제안) | 생성 코드 검토 필요 |
| **보안 취약점** | ★★★☆☆ 중간 | L2 (PR 제안 + 인간 검토) | AI 수정 오류 위험 있음 |
| **코드 스멜/리팩터링** | ★★☆☆☆ 낮음 | L2 (에이전트 제안) | 비즈니스 로직 변경 위험 |
| **파일 크기 분리** | ★★★☆☆ 중간 | L2 (분리 PR 제안) | 현재 core/telegram_relay.py 해당 |
| **반복 에러 패턴** | ★★★★☆ 높음 | L3 (자동 패치) | 패턴이 명확한 경우 |

### 3-5. 패턴별 장단점 비교

| 패턴 | 장점 | 단점 | 실패 모드 |
|------|------|------|----------|
| **임계값 트리거** | 명확한 기준, 예측 가능 | 새로운 유형 이슈 누락 | 임계값 설정 오류로 오발동/미발동 |
| **이벤트 트리거** | 실시간 대응 | 과부하 위험 | CI 루프 무한 반복 (수정 실패 반복) |
| **단방향 파이프라인** | 단순, 안전 | 자동 개선 없음 | 알림 피로(alert fatigue) |
| **반자동 루프** | 균형 | 인간 병목 | PR 미검토 누적 |
| **완전 자동 루프** | 빠름 | 보안 위험 | AI 취약점 주입, 무한 수정 루프 |

| 단계 | 역할 | 추천 도구 | 비고 |
|------|------|----------|------|
| **리포트 생성** | 코드 건강 측정 | 현재 내장 scanner | 이미 존재 |
| **트리거 판단** | 임계값/이벤트 판단 | Python 스크립트 + YAML 규칙 | 간단 구현 |
| **린팅 자동 수정** | 포맷/린트 자동 수정 | `ruff --fix` + pre-commit | 즉시 적용 가능 |
| **의존성 자동 PR** | 의존성 업데이트 | Renovate (GitHub App) | 설정 파일 추가만으로 가능 |
| **코드 수정 에이전트** | CRITICAL 이슈 수정 | Claude Code API / Dagger + LLM | 중간 난이도 |
| **PR 자동 승인/머지** | 조건부 자동 머지 | Mergify | `.mergify.yml` 설정 |
| **결과 검증** | CI 재실행 + 테스트 | 기존 GitHub Actions | 재활용 |
| **피드백 저장** | 패턴 DB 누적 | AGENTS.md / shared_memory.py | 이미 존재 |
| **알림** | 텔레그램 보고 | 기존 PM 알림 시스템 | 이미 존재 |

### 4-4. 현재 telegram-ai-org 이슈에 직접 적용 방법

**이슈 1: `core/telegram_relay.py` 205KB (크리티컬)**
- 패턴: L2 반자동 루프
- 방법: 리포트 트리거 → Claude Code 에이전트가 파일 분리 PR 생성 → PM 검토 후 머지
- 구현 난이도: 중간 (에이전트 호출 워크플로우 작성 필요)

**이슈 2: `approach` 에러 패턴 49회 반복**
- 패턴: L3 이벤트 기반 자동 패치
- 방법: 리포트에서 반복 패턴 감지 → 공통 수정 패치 자동 생성 → CI 통과 시 자동 커밋
- 구현 난이도: 낮음 (패턴이 명확한 경우 sed/ruff rule로 처리 가능)
