# 텔레그램 마크다운 파싱 스펙 조사 보고서

> 목적: 텔레그램 채널/봇 메시지 마크다운 파싱 현황 진단 및 수정 권고

---

## 1. 현황 요약

### parse_mode 현황

| 항목 | 현황 |
|------|------|
| 코드베이스 parse_mode 표준 | **100% HTML** (MarkdownV2/Markdown 0건) |
| 핵심 변환 함수 | `core/telegram_formatting.py::markdown_to_html()` |
| HTML 이스케이프 함수 | `core/telegram_formatting.py::escape_html()` |
| parse_mode 미지정 전송부 | ~35건 (대부분 고정 문자열 — LOW 위험) |

**결론:** 현재 코드베이스는 이미 올바른 방향(`parse_mode="HTML"`)으로 구현되어 있으나,
`markdown_to_html()` 변환이 누락된 채 전송되거나, 변환 로직에 커버하지 못하는 마크다운 패턴이 존재하여 렌더링이 깨지는 것으로 진단.

## 2. 텔레그램 공식 스펙 요약

### HTML vs MarkdownV2 비교

| 항목 | HTML | MarkdownV2 |
|------|------|-----------|
| 이스케이프 문자 수 | 3개 (`&` `<` `>`) | 18개 전체 |
| 프로그래밍 난이도 | 쉬움 | 어려움 (누락 시 400 에러) |
| LLM 출력 변환 적합성 | ✅ 높음 | ❌ 낮음 |
| 현재 코드베이스 사용 | ✅ HTML | 미사용 |

### HTML 지원 태그

```
<b>, <strong>               → 굵게
<i>, <em>                   → 기울임
<u>, <ins>                  → 밑줄
<s>, <strike>, <del>        → 취소선
<code>                      → 인라인 코드
<pre>                       → 코드 블록
<a href="URL">              → 링크
<blockquote>                → 인용구
<blockquote expandable>     → 펼침형 인용구
<span class="tg-spoiler">   → 스포일러
```

**HTML parse_mode가 현재 코드베이스에 적합한 선택임을 공식 스펙이 확인.**

## 3. 리스크 항목

### 3-1. 🔴 HIGH: parse_mode 미지정 + 변수 전송

| 파일 | 라인 | 내용 | 문제점 |
|------|------|------|-------|
| telegram_relay.py | 3835 | `reply_text(text_out)` | `/sessions` 결과물 — 마크다운 포함 가능하나 plain text 전송 |
| telegram_relay.py | 3903 | `reply_text(result)` | `/compact` 결과물 — LLM 응답 포함 가능성 있음 |

**영향:** 이 두 경로에서 LLM이 생성한 마크다운 텍스트가 포맷팅 없이 `**굵게**` 형태 그대로 노출될 수 있음.

### 3-2. 🟡 MEDIUM: parse_mode="HTML"이나 markdown_to_html() 미적용

| 파일 | 라인 | 내용 | 문제점 |
|------|------|------|-------|
| telegram_relay.py | 3659 | `reply_text(msg, parse_mode="HTML")` | `msg` 내 `d.get('direction', '')` 값이 `<`, `>` 포함 시 HTML 파싱 오류 |
| telegram_relay.py | 3686 | `reply_text(msg, parse_mode="HTML")` | 동일 패턴 — 사용자 입력값 포함 |
| telegram_relay.py | 4092 | `reply_text("\n".join(lines), parse_mode="HTML")` | lines 내 `&`, `<`, `>` 미이스케이프 가능 |
| telegram_relay.py | 4115 | `reply_text("\n".join(lines), parse_mode="HTML")` | 동일 패턴 |

**영향:** HTML 태그가 포함된 사용자 입력이나 시스템 데이터가 있으면 HTML 파싱 오류로 메시지 전송 실패.

### 3-3. 🟡 MEDIUM: markdown_to_html() 변환 함수 커버리지 한계

현재 `markdown_to_html()` 구현에서 커버되지 않는 패턴:

| 마크다운 패턴 | 현재 처리 | 텔레그램 HTML 결과 |
|-------------|---------|----------------|
| `__text__` (밑줄) | `<b>` 변환 (bold로 처리) | 텔레그램 밑줄(`<u>`)이 아닌 bold 렌더링 |
| `\|\|text\|\|` (스포일러) | 미지원 | 그대로 노출 |
| 순서있는 목록 `1. item` | 변환 없음 (숫자 그대로 유지) | plain text로 보임 (의도된 동작) |
| 중첩 강조 `**_text_**` | 일부 미처리 가능 | 렌더링 불완전 가능 |
| LLM이 `---` 수평선 생성 | `──────────` 변환 | 정상 (유니코드 선) |

### 3-4. 🟡 MEDIUM: E2E 스크립트 parse_mode 미지정

| 파일 | 라인 | 내용 |
|------|------|------|
| e2e_full_suite.py | 406 | `client.send_message(CHAT_ID, msg)` — parse_mode 미지정 |
| e2e_session_*.py | 274 | 동일 패턴 |

**영향:** E2E 테스트가 실제 텔레그램 채널에 보내는 메시지에서 마크다운 렌더링이 안 됨.
(단, 이는 Telethon/Pyrogram 클라이언트이므로 별도 parse_mode 지정 방식이 다를 수 있음)

## 4. 권고 조치

### 우선순위 1 (즉시 수정) — HIGH 리스크 2건

```python
# telegram_relay.py L3835
# 수정 전
await update.message.reply_text(text_out)
# 수정 후
await update.message.reply_text(markdown_to_html(text_out), parse_mode="HTML")

# telegram_relay.py L3903
# 수정 전
await update.message.reply_text(result)
# 수정 후
await update.message.reply_text(markdown_to_html(result), parse_mode="HTML")
```

### 우선순위 2 (단기 수정) — MEDIUM 리스크 4건 (HTML 수동 조립 + escape 누락)

```python
# telegram_relay.py L3659, 3686 패턴
# 사용자 입력값(d.get('direction', ''))을 HTML에 삽입 시 escape_html() 적용
direction = escape_html(d.get('direction', '미설정'))

# telegram_relay.py L4092, 4115 패턴
# lines 를 "\n".join 전에 각 항목 escape_html 처리
lines = [escape_html(line) for line in raw_lines]
await update.message.reply_text("\n".join(lines), parse_mode="HTML")
```

### 우선순위 3 (중기 개선) — markdown_to_html() 보완

1. **밑줄 지원 추가**: `__text__` → `<u>text</u>` (현재 bold로 처리 중)
   - 단, LLM이 밑줄용으로 `__` 를 쓰는 경우 rare → 낮은 우선순위
2. **스포일러 지원**: `||text||` → `<span class="tg-spoiler">text</span>`
3. **중첩 강조 처리 개선**: `**_text_**` 케이스 테스트 커버리지 추가

### parse_mode 표준 선택 권고: **HTML 유지**

- 이미 채택 중이며 올바른 선택
- MarkdownV2로 전환 시 18개 특수문자를 모두 이스케이프해야 해서 득보다 실이 큼
- LLM이 생성하는 마크다운을 HTML로 변환하는 현재 아키텍처(`markdown_to_html()`)가 최선
