# Retro — Gotchas

이 스킬을 사용할 때 자주 발생하는 실수와 주의사항이다.

## Gotcha 1: BLOCKER 분석에서 5 Whys를 1-2 Whys로 중단
**상황**: 근본 원인 분석 시 "왜?" 를 충분히 반복하지 않고 표면적 원인에서 멈출 때
**증상**: "봇이 응답 느렸음 → API 타임아웃 때문" 에서 분석 종료. 근본 원인(재시도 로직 부재, 연결 풀 미설정)이 식별되지 않아 다음 스프린트에 동일 장애 재발
**해결**: BLOCKER 항목마다 최소 5번 "왜?" 를 반복하는 구조 강제. 마지막 "왜?" 의 답이 코드/설정/프로세스 중 하나로 귀결될 때까지 진행. 5번 이전에 "제어 불가 외부 요인"이 나오면 해당 시점에서 종료하고 이유 명시

## Gotcha 2: 액션 아이템을 담당자 없이 배분
**상황**: 개선 액션 아이템을 "팀 전체"나 담당자 미지정 상태로 도출할 때
**증상**: 모두의 할 일이 아무도의 할 일이 됨. 다음 회고에서 동일 액션 아이템이 "미이행"으로 재등장
**해결**: 액션 아이템 형식 강제 — "담당: [봇명] / 기한: [날짜] / 완료 기준: [측정 가능한 결과]". telegram-ai-org 에서는 봇명이 명확하므로 반드시 봇 단위로 배분

## Gotcha 3: CLAUDE.md 업데이트를 "권장"으로 처리하고 실제 반영 안 함
**상황**: Step 3에서 CLAUDE.md 운영 주의사항 업데이트를 필요한 것으로 식별했지만 실제 파일 수정 없이 "권장 사항" 으로만 기록할 때
**증상**: 회고에서 발견한 레슨이 다음 에이전트 세션에 전달되지 않음. 같은 실수가 다음 스프린트에 반복
**해결**: Step 3 완료 조건: CLAUDE.md `운영 주의사항` 섹션에 새 레슨을 실제로 추가(Write/Edit 실행). 추가 후 git diff로 변경 확인. 회고 보고서에 "CLAUDE.md 업데이트: [추가된 항목]" 명시

## Gotcha 4: 회고 대상 기간의 데이터 없이 인상 기반 회고 진행
**상황**: 실제 run 이력, 로그, 태스크 기록을 확인하지 않고 에이전트의 기억에 의존해 Start/Stop/Continue를 작성할 때
**증상**: 실제로 자주 발생한 문제가 누락되고, 기억에 남는 단발 이슈가 과대 대표됨. 데이터 기반이 아닌 인상 기반 회고로 액션 아이템의 실효성 저하
**해결**: Step 1(봇 회고 요청) 전에 반드시 `.ai-org/runs/`, `logs/`, `docs/retros/` 에서 해당 기간 데이터 수집. 구체적 수치(장애 횟수, 완료 태스크 수, 평균 처리 시간)를 회고 프롬프트에 포함

## Gotcha 5: 회고 보고서가 저장되지 않아 이전 회고 이력 단절
**상황**: `docs/retros/YYYY-MM-DD-retro.md` 저장 전 에이전트가 종료되거나 경로 오류 발생 시
**증상**: 회고 내용이 대화 컨텍스트에만 존재하다 사라짐. 다음 회고 시 "이전에도 같은 문제가 있었는가?" 확인 불가. 패턴 분석 단절
**해결**: 회고 보고서 저장을 Step 2(패턴 분석) 직후, Step 3(액션 배분) 이전에 실행. 중간 저장으로 내용 유실 방지. 저장 후 Rocky에게 파일 경로 보고
