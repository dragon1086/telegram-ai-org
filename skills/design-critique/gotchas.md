# Design Critique — Gotchas

이 스킬을 사용할 때 자주 발생하는 실수와 주의사항이다.

## Gotcha 1: Telegram UI에 WCAG 기준 기계적 적용
**상황**: Telegram 봇 메시지 출력물(텍스트, 인라인 버튼)에 웹 WCAG 2.1 AA 기준을 그대로 적용할 때
**증상**: "색상 대비 4.5:1 미충족" 같은 지적이 나오지만 Telegram 클라이언트가 색상을 제어하므로 개선 불가. 실제 수정 가능한 문제(버튼 텍스트 모호성, 긴 메시지 가독성)는 놓침
**해결**: 검토 대상이 Telegram 봇 출력인지 웹 UI인지 먼저 구분. 봇 출력은 WCAG 대신 "메시지 명확성, 버튼 라벨 직관성, 오류 메시지 회복 안내" 기준으로 평가

## Gotcha 2: 피드백 심각도 기준 없이 모두 "높음" 표기
**상황**: 피드백 작성 시 심각도(높음/중간/낮음) 판단 근거 없이 분류할 때
**증상**: 사소한 오탈자와 접근성 치명 결함이 같은 "높음"으로 표기됨. 개선 우선순위를 정하지 못하고 모든 피드백이 무시되거나 전부 처리하려다 지연 발생
**해결**: 심각도 기준 명시 — 높음: 기능 차단/데이터 손실/보안, 중간: 사용성 저하/빈번한 오류, 낮음: 심미적/선호도 문제. 각 피드백에 이 기준 적용 근거 한 줄 추가

## Gotcha 3: 모바일 검토를 데스크톱 스크린샷으로 대체
**상황**: 반응형 동작 확인 시 실제 모바일 환경 없이 데스크톱 화면만 검토할 때
**증상**: Telegram 앱에서 긴 인라인 버튼 텍스트가 잘리는 문제, 모바일에서 메시지 줄바꿈 위치 이상 등을 발견 못 함
**해결**: Telegram 봇의 경우 메시지당 글자 수 제한(4096자), 버튼 텍스트 길이(최대 64자), 인라인 키보드 열 수(최대 8열) 기준으로 구조적 검토. 실제 디바이스 테스트 없을 때는 이 수치 기준으로 대체

## Gotcha 4: 디자인 시스템 부재 프로젝트에서 "일관성" 지적만 반복
**상황**: 공식 디자인 시스템이 없는 프로젝트에서 일관성 검토를 수행할 때
**증상**: "버튼 스타일이 일관되지 않음" 지적을 남기지만 참조할 기준이 없어 개선 방향을 제시 못 함. 피드백이 실행 불가능한 상태로 종료
**해결**: 디자인 시스템 부재 확인 시 먼저 현재 패턴에서 사실상 표준(de facto standard)을 추출하여 기술. 그 기준 대비 이탈 사례를 지적하는 순서로 진행
