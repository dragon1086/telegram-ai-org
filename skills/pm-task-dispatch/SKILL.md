# PM Task Dispatch (업무배분 스킬)

PM 봇이 수신된 태스크를 분석하고 가장 적합한 조직 봇에 배분하는 체계적 프로세스.

---

## Step 0: 태스크 유형 자율 분류 (배분 전 필수)

배분 메시지를 작성하기 전, LLM은 아래 판단 체인을 스스로 실행한다.

### 판단 체인 (Chain-of-Thought)

**Q1. 이 태스크의 최종 산출물은 무엇인가?**
- 코드 / 파일 생성 / DB 변경 / 설정 변경 → **실행형** 그룹 (구현 · 수정 · 운영)
- 문서 / 보고서 / 분석표 / 기획서 → **사고형** 그룹 (조사 · 분석 · 기획 · 설계 · 검토)

**Q2. 요청 문장의 핵심 동사는 무엇인가?**

| 동사 신호 | 유형 | 파일 변경 |
|-----------|------|-----------|
| 만들어 / 구현해 / 코딩해 / 개발해 | 구현 | ✅ 예 |
| 고쳐 / 수정해 / 개선해 / 바꿔 / 업데이트해 | 수정 | ✅ 예 |
| 배포해 / 재시작 / 서버 / 인프라 / 모니터링 | 운영 | ✅ 예 |
| 조사해 / 찾아봐 / 알아봐 / 레퍼런스 | 조사 | ❌ 아니오 |
| 분석해 / 비교해 / 평가해 / 왜 그런지 | 분석 | ❌ 아니오 |
| 기획해 / PRD / 요구사항 정의 / 설계 방향 | 기획 | ❌ 아니오 |
| 설계해 / 아키텍처 / 구조 잡아 / 와이어프레임 | 설계 | ❌ 아니오 |
| 봐줘 / 검토해 / 리뷰해 / 어때 / 문제 없어 | 검토 | ❌ 아니오 |

**Q3. 파일·코드를 직접 변경해야 하는가?**
- YES → 실행형 유형 재확인 후 `파일 변경 허용: 예` 명시
- NO → 사고형 유형 재확인 후 `파일 변경 허용: 아니오` 명시

**Q4. 유형 확신도가 60% 미만인가?**
→ 강제로 단일 유형 결정하지 말 것. 아래 Fallback 정책 적용.

### 유형별 뉘앙스 차이 요약

| 유형 | 핵심 뉘앙스 | 전형적 오분류 위험 |
|------|------------|-----------------|
| **조사** | "뭐가 있는지" 사실 수집. 해석 없음 | 분석으로 확장 → 보고서에 구현 제안 삽입 |
| **분석** | "왜/어떻게/무엇이 나은지" 해석. 코드 없음 | 분석 결과를 바로 구현 코드로 변환 |
| **기획** | "무엇을 만들지" 결정. PRD/요구사항 산출 | PRD 안에 코드 스니펫·파일 편집 지침 삽입 |
| **설계** | "어떻게 만들지" 구체화. 다이어그램/명세 | 설계서를 작성하다가 코드 구현까지 진행 |
| **검토** | "이게 맞나" 판단. 기존 산출물 대상 | 리뷰 코멘트 수준 초과해 직접 수정 |
| **수정** | 기존 것 일부 변경. 범위 엄격히 제한 | 수정 중 관련 없는 파일 추가 개선 |
| **구현** | 새로 만들기. 코드·파일 생성 포함 | 구현 완료 후 테스트·배포까지 자동 진행 |
| **운영** | 인프라·배포·재시작. infra 역할 봇 전용 | 다른 봇이 직접 restart/deploy 실행 |

### Fallback 정책 (불확실 시)

```
확신도 < 60% → 아래 중 하나 선택:
  1. 태스크를 "조사" → "기획" → "구현" 단계로 분리해 순서 배분
  2. 사고형으로 먼저 배분하고 산출물 검토 후 다음 단계 결정
  3. 요청 문장에 동사가 없거나 모호한 경우: 기본값 = "분석" (파일 변경 없음)

절대 금지:
  → "혼합이니까 다 해도 된다"는 판단 — 범위 무제한 허용 아님
  → 분석 요청에 구현 착수
  → 검토 요청에 직접 파일 수정
```

## Step 1: 태스크 분석

태스크를 받으면 다음을 파악한다:
- **주요 영역**: 개발/디자인/성장/운영/연구/제품 중 하나 또는 복합
- **긴급도**: 즉시/오늘중/이번주
- **의존성**: 다른 봇의 산출물이 먼저 필요한지
- **예상 규모**: S(1-2h) / M(반나절) / L(하루+)

---

## Step 2: 봇 선정

```
유형 × 영역 → 담당 봇 (실제 org ID는 organizations.yaml 참조)

[사고형]
조사·분석(시장/경쟁/레퍼런스) → 리서치 역할 조직
분석·조사(데이터/지표)         → 성장/마케팅 역할 조직
기획·설계(PRD/요구사항)        → 제품/기획 역할 조직
설계(아키텍처/시스템 구조)      → 개발/엔지니어링 역할 조직
설계(UI/UX/와이어프레임)        → 디자인 역할 조직
검토(코드/설계 리뷰)            → 원래 산출물을 만든 담당 봇

[실행형]
구현·수정(코드/API/버그)       → 개발/엔지니어링 역할 조직
구현·수정(UI/디자인)           → 디자인 역할 조직
운영(인프라/배포/재시작)        → infra 역할 조직 (capabilities: [infra])

[복합]
복합 태스크 → 주담당 봇 배분 + 나머지는 별도 서브태스크로 분리
```

## Step 3: 배분 메시지 작성

```
[태스크 배분] #{태스크ID}
담당: @{봇이름}
태스크 유형: {조사|분석|기획|설계|검토|수정|구현|운영}
판단 근거: {핵심 동사/의도 한 줄 — 예: "분석해줘" → 해석 산출, 파일 변경 없음}
파일·코드 변경 허용: {예 / 아니오}
내용: {태스크 설명}
긴급도: {즉시/오늘중/이번주}
기대 산출물: {구체적 결과물 — 형식·분량 포함}
실행 범위: {이 봇이 해야 할 것만 명시 — 범위 외 작업은 별도 서브태스크}
협업 필요: {다른 봇 있으면 명시, 없으면 생략}
```

> **파일·코드 변경 허용: 아니오** 인 태스크에서 봇이 파일을 수정·생성·삭제하면 즉시 오버스콥.
> PM은 이 필드를 보고 봇의 결과물을 검수해야 한다.

> **PM 스코프 원칙**: 각 봇은 배분받은 태스크의 "실행 범위"에 명시된 것만 수행한다.
> 범위에 없는 작업(재기동, push, merge 등)은 PM이 별도 태스크로 분리해 적절한 봇에 배분해야 한다.
> 봇 스스로 범위를 확장하지 않는다 — PM이 명확히 정의하는 것이 책임이다.

## Step 4a: 복수 조직 결과 통합 시 불일치 감지

PM이 여러 조직의 산출물을 받아 통합할 때:

```
불일치 체크리스트:
- [ ] 동일 항목(라이선스·버전·수치·결론)에 대해 조직간 상반된 결과 없는가?
- [ ] 있다면: pm-discussion 스킬 즉시 발동
- [ ] 없다면: 통합 보고서 작성 진행
```

불일치 발견 시 pm-discussion 발동 형식:
```
/pm-discussion "불일치 해소 필요: {항목명}"
조직A 결론: {요약}
조직B 결론: {요약}
→ 어느 쪽이 맞는지 각 조직은 근거와 출처를 제시하라.
```

> ⚠️ 불일치를 발견하고도 pm-discussion 없이 "둘 다 나열"하는 보고서는 금지.
> PM은 결론을 확정하거나 "미확정 + 이유"를 사용자에게 명시해야 한다.

## Step 4: 진행 추적
- 배분 후 `orchestration.yaml` docs_root에 태스크 상태 기록
- 완료 예상 시간이 지나면 팔로업 메시지 전송
- 완료 보고 수신 시 Rocky에게 요약 전달

## 복합 태스크 처리

여러 봇이 필요한 태스크:
1. 의존성 그래프 분석
2. **자기파괴 작업 분리**: restart/deploy/reboot/재기동이 포함되면 반드시 별도 서브태스크로 분리
3. 병렬 실행 가능한 부분 식별
4. 순서대로 배분 (병렬 가능한 것은 동시 배분)

> ⚠️ 실행 주체와 kill 대상이 겹치는 작업은 절대 같은 태스크에 넣지 않는다.
> 예: engineering_bot이 코드 수정 + restart_bots.sh 실행 → 자기 자신 kill → 무한 루프

## Prerequisites

배분 대상이 코드 변경 또는 배포를 포함하는 경우, **quality-gate 스킬을 먼저 실행**하라.

```
적용 기준:
- 태스크 유형이 [구현] 또는 [수정]인 경우
- 배포(deploy) 또는 병합(merge)이 포함된 경우
- 개발실 봇에 배분하기 전 반드시 quality-gate 통과 확인

실행 순서:
1. quality-gate 스킬 실행 → PASS 확인
2. PASS 시에만 pm-task-dispatch Step 1 진행
3. FAIL 시 → 개발실에 수정 요청 후 재검사
```

## 스코프 분리 원칙 (글로벌 규칙)

각 봇은 자기 전문 영역의 일만 한다. PM이 위임할 때 이를 명시적으로 지정해야 한다.

| 작업 유형 | 담당 봇 | 다른 봇에서 요청 시 |
|-----------|---------|-------------------|
| 재기동 / restart | infra 역할 조직만 | COLLAB 요청 금지, PM이 별도 태스크 |
| git push / merge to main | infra 역할 조직만 | COLLAB 요청 금지, PM이 별도 태스크 |
| 코드 수정 / 버그 수정 | 개발/엔지니어링 역할 조직 | - |
| 시장조사 / 레퍼런스 | 리서치 역할 조직 | - |
| 배포 / 인프라 변경 | infra 역할 조직 | - |

> 봇별 프롬프트에 금지 문구를 박는 것보다, **PM이 위임 시 범위를 명확히 정하는 것**이 글로벌하게 작동한다.

## 부서 간 충돌 감지 → 자동 조율 태스크 생성

결과 합성 시 `CONFLICTING` 판정이 나면 PM은 수동 개입 없이 아래 흐름을 자동 실행한다.

### 자동 조율 흐름 (트리거 → 탐지 → 태스크 생성 → 알림)

```
[트리거] 부서 결과 수집 완료 + ResultSynthesizer.synthesize() 호출
    ↓
[탐지]  synthesis.judgment == CONFLICTING
    ↓
[알림]  "⚠️ 부서 간 결과 충돌 감지" 메시지 (reasoning 포함)
    ↓
[생성]  synthesis.follow_up_tasks → SubTask 변환 → dispatch()
    ↓  (LLM이 follow_up 미제공 시 fallback)
[fallback] product_bot → research_bot 순서로 기본 조율 태스크 자동 생성
    ↓
[알림]  "🔀 충돌 자동 조율 태스크 N건 생성 중..."
```

### 조율 태스크 표준 필드

충돌 조율 태스크 설명에는 반드시 아래 필드가 포함되어야 한다.

| 필드 | 설명 | 예시 |
|------|------|------|
| `충돌 사유` | ResultSynthesizer가 탐지한 reasoning | "MIT vs Apache 2.0 라이선스 불일치" |
| `충돌 유형` | 결과 불일치 / 팩트 충돌 / 우선순위 충돌 / 의존성 기한 역전 / 스코프 중복 | "팩트 충돌" |
| `관련 부서` | 충돌에 관여한 조직 org_id 목록 | "aiorg_engineering_bot, aiorg_research_bot" |
| `우선순위` | high / medium / low | "high" |
| `기한` | 즉시 / 오늘중 / 이번주 | "즉시" |
| `원본 요청` | 사용자 최초 요청 (300자 이내) | — |

### 감지 케이스 (5종)

| 케이스 | 설명 | 기본 조율 담당 |
|--------|------|---------------|
| **팩트 충돌** | 같은 데이터(버전/라이선스/수치)를 서로 다르게 보고 | research_bot |
| **방향 불일치** | 아키텍처·기술 선택에서 상반된 권고 | product_bot |
| **우선순위 충돌** | 동일 리소스 이중 배정 또는 중요도 판단 차이 | product_bot |
| **의존 기한 역전** | A 완료 전 B가 먼저 완료 불가한 순서 문제 | product_bot |
| **스코프 중복** | 두 부서가 동일 산출물을 각자 생성 | product_bot |

> **구현 위치**: `core/pm_orchestrator.py` CONFLICTING 분기, `core/result_synthesizer.py` CONFLICTING RULE 프롬프트

## 안티패턴

- ❌ 모든 태스크를 개발실에만 배분
- ❌ 의존성 고려 없이 동시 배분
- ❌ quality-gate 없이 코드 변경 배분
- ❌ 코드 작업과 restart/deploy를 같은 태스크로 배분 (자기파괴 루프 위험)
- ❌ 태스크 유형 없이 배분 → 봇이 스스로 범위 결정
- ❌ "혼합이니까 다 해도 된다" → 사실상 무제한 허용
- ❌ 파일 변경 허용 필드 생략 → 봇이 임의로 파일 수정
- ❌ CONFLICTING 판정 후 조율 태스크 미생성
- ✅ Step 0 판단 체인 실행 후 태스크 유형 명시
- ✅ 판단 근거 한 줄 작성 (오분류 자가검증)
- ✅ 파일·코드 변경 허용 여부 명시
- ✅ 태스크 성격에 맞는 봇 선정
- ✅ 배분 메시지에 "실행 범위" 항목 명시
- ✅ 코드/배포 태스크는 quality-gate 먼저
- ✅ restart/deploy는 ops_bot 별도 서브태스크

## 신규 봇 온보딩 체크리스트

새 조직(봇)을 추가할 때 아래 항목을 반드시 확인한다:

- [ ] `organizations.yaml`에 등록 (capabilities 포함)
- [ ] `orchestration.yaml` → `global_instructions` 자동 적용 확인 (별도 per-org 스코프 문구 추가 불필요)
- [ ] `bots/{org_id}.yaml` 생성 (role, instruction, team_config 포함)
