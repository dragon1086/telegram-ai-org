# PM Task Dispatch — Gotchas

## 1. 복합 태스크의 의존성 순서
"디자인 후 개발" 같은 태스크는 반드시 순서대로 배분.
병렬 실행 가능 여부를 먼저 판단하고, 의존성이 있으면 순차 배분.

## 2. 기본 담당봇 오판
"API 문서 작성"은 개발실이 아닌 제품실(기획)이 더 적합할 수 있음.
키워드 매칭만으로 판단하지 말고 태스크 전체 맥락을 읽을 것.

## 3. 배분 후 추적 누락
배분했다고 끝이 아님. 완료 예상 시간 후 팔로업 메시지 필수.
`config.json`의 `default_timeout_hours` 기준으로 체크.

## 4. chat_id 하드코딩 주의
`config.json`의 `chat_id`를 수정 없이 사용하면 잘못된 그룹에 메시지 전송.
실행 전 반드시 `TELEGRAM_GROUP_CHAT_ID` 환경변수 확인.

## 5. 봇 오프라인 상태 미확인
배분 전 봇이 응답 가능한지 확인 안 하면 태스크가 사라짐.
`/harness-audit` 스킬로 봇 상태 먼저 점검 권장.

## 6. 자기파괴 작업(restart/deploy)을 코드 작업과 합치지 말 것
**사고 사례**: T-224 "마크다운 수정 + 커밋 + 봇 재시작"을 engineering_bot 단일 태스크로 배분
→ 봇이 코드 수정 후 restart_bots.sh 실행 → 자기 자신 kill → lease release 안 됨
→ 재시작 후 동일 태스크 재실행 → 무한 루프 (3회 커밋 반복)

**규칙**: restart, deploy, reboot, 재기동 키워드가 포함된 태스크는 반드시 별도 서브태스크로 분리.
```
❌ "코드 수정 + 커밋 + 봇 재시작" → 하나의 태스크
✅ T-sub1: "코드 수정, 커밋, 푸시" → engineering_bot
   T-sub2: "봇 전체 재시작" → ops_bot (T-sub1 의존성)
```
코드 작업 봇이 자기 자신을 죽이는 명령을 실행하면 send_report도 lease release도 호출 불가.
실행 주체와 kill 대상이 겹치지 않도록 배분할 것.

## 7. 봇 프롬프트에 과적합 금지 규칙 박지 말 것
**사고 사례**: T-238~240 에서 개발실·리서치실 bot yaml에 "재기동 금지" 문구를 직접 삽입
→ 개발실·리서치실만 제한되고 다른 부서에는 미적용 → 일관성 없는 행동
→ 운영실 bot yaml에는 "단독 권한" 부여 → 다른 조직들과 비대칭 규칙 난립

**원인**: 봇별 프롬프트에 서로 다른 금지/허용 규칙을 박으면 유지보수 불가능

**규칙**: 스코프 제한은 봇 프롬프트가 아니라 **PM 배분 메시지의 "실행 범위" 항목**으로 정의한다.
```
❌ bots/aiorg_engineering_bot.yaml guidance에 "재기동 금지" 문구 삽입
✅ PM이 태스크 배분 시 "실행 범위: 코드 수정·커밋만. 재기동 제외." 명시
```
PM이 명확한 범위를 지정하면 모든 봇에 글로벌하게 적용된다.

## 8. 태스크 의존성 등록 전 TaskPoller 조기 수신 — 병렬 실행 레이스 컨디션
**사고 사례**: T-237 작업에서 PM이 리서치→개발실→운영실 순서를 의도했지만 3개 태스크가 병렬 실행됨.
→ `_dispatch_subtasks` Step 1(태스크 생성) 중 LLM 프롬프트 생성에 ~17–20s 소요
→ T-239(개발실)가 DB에 생성되었으나 deps(T-239→T-238)는 아직 미등록 상태
→ 개발실 TaskPoller(2s 주기)가 T-239를 deps 없는 즉시 실행 가능 태스크로 인식하여 선점
→ 결과: 리서치(T-238)와 개발실(T-239)·운영실(T-240) 병렬 실행. 개발실이 리서치보다 먼저 완료됨.

**원인**: 태스크 생성(Step 1) → 의존성 등록(Step 2) → 첫 웨이브 발송(Step 3) 순서에서
의존성 등록 전에 이미 TaskPoller가 'pending' 태스크를 수신할 수 있었음.

**수정**: `pm_orchestrator._dispatch_subtasks` 를 "ID 사전 생성 + deps 먼저 등록 → 태스크 생성" 순서로 변경 (commit: fix/dep-race).
```
❌ 이전: 태스크 생성 루프(LLM ~20s/개) → 의존성 등록 (늦게 등록됨)
✅ 수정: 태스크 ID 생성(빠름) → 의존성 먼저 등록 → 태스크 생성 루프(LLM)
```
이제 T-239가 생성될 때 이미 pm_task_dependencies에 T-239→T-238이 있으므로 TaskPoller가 T-238 완료 전에 T-239를 수신하지 않는다.
