# PM Task Dispatch — Gotchas

## 1. 복합 태스크의 의존성 순서
"디자인 후 개발" 같은 태스크는 반드시 순서대로 배분.
병렬 실행 가능 여부를 먼저 판단하고, 의존성이 있으면 순차 배분.

## 2. 기본 담당봇 오판
"API 문서 작성"은 개발실이 아닌 제품실(기획)이 더 적합할 수 있음.
키워드 매칭만으로 판단하지 말고 태스크 전체 맥락을 읽을 것.

## 3. 배분 후 추적 누락
배분했다고 끝이 아님. 완료 예상 시간 후 팔로업 메시지 필수.
`config.json`의 `default_timeout_hours` 기준으로 체크.

## 4. chat_id 하드코딩 주의
`config.json`의 `chat_id`를 수정 없이 사용하면 잘못된 그룹에 메시지 전송.
실행 전 반드시 `TELEGRAM_GROUP_CHAT_ID` 환경변수 확인.

## 5. 봇 오프라인 상태 미확인
배분 전 봇이 응답 가능한지 확인 안 하면 태스크가 사라짐.
`/harness-audit` 스킬로 봇 상태 먼저 점검 권장.

## 6. 자기파괴 작업(restart/deploy)을 코드 작업과 합치지 말 것
**사고 사례**: T-224 "마크다운 수정 + 커밋 + 봇 재시작"을 engineering_bot 단일 태스크로 배분
→ 봇이 코드 수정 후 restart_bots.sh 실행 → 자기 자신 kill → lease release 안 됨
→ 재시작 후 동일 태스크 재실행 → 무한 루프 (3회 커밋 반복)

**규칙**: restart, deploy, reboot, 재기동 키워드가 포함된 태스크는 반드시 별도 서브태스크로 분리.
```
❌ "코드 수정 + 커밋 + 봇 재시작" → 하나의 태스크
✅ T-sub1: "코드 수정, 커밋, 푸시" → engineering_bot
   T-sub2: "봇 전체 재시작" → ops_bot (T-sub1 의존성)
```
코드 작업 봇이 자기 자신을 죽이는 명령을 실행하면 send_report도 lease release도 호출 불가.
실행 주체와 kill 대상이 겹치지 않도록 배분할 것.
