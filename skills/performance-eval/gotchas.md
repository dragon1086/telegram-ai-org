# Performance Eval — Gotchas

이 스킬을 사용할 때 자주 발생하는 실수와 주의사항이다.

## Gotcha 1: .ai-org/runs/ 분석 시 미완료 run을 "실패"로 집계
**상황**: `.ai-org/runs/` 디렉토리에서 봇별 완료율을 계산할 때 완료 마커 없는 run을 모두 실패로 처리할 때
**증상**: 현재 진행 중인 run, 인터럽트된 후 재개 예정인 run이 모두 실패로 집계됨. 봇의 실제 완료율이 70%인데 40%로 보고되어 부당한 평가
**해결**: 완료율 계산 기준 명확화 — 완료: 완료 마커 있는 run, 실패: 24시간 이상 경과 후 완료 마커 없는 run, 진행 중: 최근 24시간 이내 타임스탬프. "진행 중" run은 완료율 분모에서 제외

## Gotcha 2: 재작업 횟수를 "품질 낮음"과 동일시
**상황**: 코드 리뷰 통과율 계산 시 재작업 횟수가 많은 봇을 낮은 품질로 단순 판단할 때
**증상**: 복잡하고 어려운 태스크를 담당한 봇이 쉬운 태스크를 담당한 봇보다 낮은 품질 점수를 받음. 태스크 난이도가 반영되지 않아 평가 불공정
**해결**: 재작업 횟수는 태스크 복잡도(LOC 변경량, 파일 수)로 정규화. 동일 태스크 유형 내에서만 봇 간 비교. 평가 보고서에 "태스크 복잡도 지수" 컬럼 추가

## Gotcha 3: "학습" 지표를 같은 에러 로그 반복으로만 측정
**상황**: 학습 지표(같은 실수 반복 여부)를 단순히 동일한 에러 메시지 반복으로 측정할 때
**증상**: 에러 메시지는 다르지만 동일한 패턴의 실수(예: async 함수에서 blocking 호출)를 반복하는 경우를 학습으로 착각. 또는 동일 에러가 다른 원인으로 발생한 경우를 무학습으로 오판
**해결**: 학습 지표는 CLAUDE.md `운영 주의사항`에 기록된 패턴과 대조. 해당 패턴이 이번 기간에 재발했는지 확인. 에러 메시지가 아닌 실패 패턴 유형으로 분류

## Gotcha 4: 협업 횟수를 메시지 수로 측정
**상황**: 협업 지표 계산 시 봇 간 주고받은 Telegram 메시지 수를 협업 지표로 사용할 때
**증상**: 단순 알림 전송도 협업으로 집계됨. 반면 pm-discussion을 통한 실질적 의사결정 기여는 건수로 잡기 어려움. 메시지 많은 봇이 협업 점수 독점
**해결**: 협업 지표는 pm-discussion 참여(발언 횟수), 타 봇 태스크에 대한 코드 기여(PR/커밋), shoutout 시스템 기록을 기반으로 측정. 단순 알림 메시지는 협업 집계에서 제외
