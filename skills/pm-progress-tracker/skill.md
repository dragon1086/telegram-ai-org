# PM Progress Tracker (진척관리 스킬)

PM 봇이 사용자로부터 큰 목표를 받으면, 이 스킬의 4단계 루프를 통해 완료까지 추적한다.

---

## 전체 플로우

```
[사용자 목표 수신]
       ↓
[Step 1] 목표 문서화 — memory/pm_progress_guide.md에 등록
       ↓
[Step 2] 태스크 분해 — 이터레이션 단위로 서브태스크 생성
       ↓
[Step 3] 이터레이션 루프 실행 — 부서 배분 → 완료 대기 → 품질 검증
       ↓
[완료 조건 충족?] → YES → [Step 4] 완료 처리 & 사용자 보고
                   → NO  → 미달 항목 재배분 후 Step 3 반복
```

---

## Step 1: 목표 문서화

사용자로부터 큰 목표(또는 여러 이터레이션이 필요한 작업)를 받으면 즉시 `memory/pm_progress_guide.md`에 등록한다.

### 등록 시 필수 필드

| 필드 | 설명 | 예시 |
|------|------|------|
| `목표명` | 간결한 제목 | "오픈소스화 — 원클릭 풀셋팅 패키징" |
| `담당부서` | 주담당 + 협업부서 | "개발실 (주), 운영실, 디자인실" |
| `시작일` | YYYY-MM-DD | "2026-03-24" |
| `완료조건` | 구체적이고 검증 가능한 기준 | "setup.sh 실행 후 10분 내 봇 응답, E2E PASS" |
| `현재상태` | TODO / IN_PROGRESS / DONE / BLOCKED | "IN_PROGRESS" |
| `이터레이션 로그` | 이터레이션별 완료 항목·잔여 항목 | 아래 참고 |

---

## Step 2: 태스크 분해

목표를 이터레이션 단위로 서브태스크로 분해한다. 각 이터레이션은 1~2일 분량.

```
분해 기준:
- 각 서브태스크는 단일 부서가 단독으로 완료 가능한 크기
- 의존성이 있는 태스크는 순차 배치
- 병렬 가능한 태스크는 같은 이터레이션에 배치
- 완료 기준(Definition of Done)을 서브태스크마다 명시
```

---

## Step 3: 이터레이션 루프

### 루프 트리거 조건

아래 조건 중 하나라도 충족되면 새 이터레이션을 시작한다:

| 트리거 | 설명 |
|--------|------|
| 새 목표 등록 | 사용자가 새로운 큰 목표 전달 |
| 이전 이터레이션 완료 | 현재 배분 서브태스크 모두 완료 |
| 블로커 해소 | BLOCKED 상태였던 태스크의 블로커가 제거됨 |
| 사용자 재지시 | "다시 iter 돌아", "계속해", "재개해" 등 |
| 크론 트리거 | `daily_goal_pipeline` 발동 시 미완료 목표 자동 재개 |

### 루프 내부 흐름

```
[이터레이션 N 시작]
1. pm_progress_guide.md 에서 IN_PROGRESS 목표의 잔여 서브태스크 목록 조회
2. 우선순위 순으로 배분 (pm-task-dispatch 스킬 실행)
3. 각 부서 완료 보고 수신 대기
4. 완료 보고 수신 시 → quality-gate 스킬 실행 (코드 변경이 포함된 경우)
5. quality-gate PASS → 해당 서브태스크 DONE 처리
6. quality-gate FAIL → 개발실에 수정 요청 후 재루프
7. 모든 서브태스크 DONE → 완료 조건 검증 (Step 4)
   아직 잔여 서브태스크 존재 → Step 3 반복
```

### 상태 정의

| 상태 | 의미 | 다음 액션 |
|------|------|-----------|
| `TODO` | 아직 시작 안 됨 | 이터레이션에서 배분 대상 |
| `IN_PROGRESS` | 배분 완료, 작업 중 | 완료 보고 대기 |
| `DONE` | 완료 + quality-gate PASS | 로그 기록 후 다음 태스크 |
| `BLOCKED` | 의존성/외부 요인으로 중단 | 블로커 기록, 다른 태스크로 우회 |

---

## Step 4: 완료 처리

### 완료 조건 검증

목표 등록 시 정의한 `완료조건`을 기준으로 최종 검증한다.

```
검증 체크리스트:
- [ ] 모든 서브태스크가 DONE 상태
- [ ] quality-gate 스킬 통과 (코드 포함 목표)
- [ ] 사용자가 정의한 완료조건 충족 (수치/기능 기준)
- [ ] 잔여 알려진 이슈 없음 (BLOCKED 항목 해소 또는 명시적 제외 처리)
```

### 완료 보고 형식

```
## 목표 완료 보고
목표명: {목표명}
완료일: {YYYY-MM-DD}
총 이터레이션: {N}회
완료 서브태스크: {N}개
잔여/이월 항목: {없음 또는 목록}

## 완료된 주요 산출물
- {산출물 1}
- {산출물 2}

## 다음 목표 제안 (선택)
- {사용자 목표 목록의 다음 항목}
```

## 미진 시 재처리 규칙

완료 조건 미충족 시 PM은 **자동으로** 다음 이터레이션을 시작한다.

```
미진 판정 기준:
- quality-gate FAIL
- 완료조건에서 정의한 수치 미달
- BLOCKED 태스크가 전체의 20% 초과

재처리 절차:
1. 미완료 항목만 추출하여 새 이터레이션 태스크로 등록
2. 블로커가 있으면 블로커 해소 태스크를 최우선 배분
3. 사용자에게 "이터레이션 N 완료, N+1 시작" 상태 보고
4. 사용자 확인 없이 자동 재개 (자율 실행 원칙)
```

> ⚠️ "완료됐나요?" 확인 질문 금지 — PM이 완료 조건 기준으로 스스로 판단한다.
> ⚠️ 잔여 항목이 있으면 보고 후 즉시 다음 이터레이션 시작.

## Prerequisites

- `memory/pm_progress_guide.md` — 목표 등록 템플릿 (이 스킬보다 먼저 읽을 것)
- `skills/pm-task-dispatch/skill.md` — 서브태스크 배분 규칙
- `skills/quality-gate/` — 코드 변경 포함 이터레이션의 완료 검증

---

## 안티패턴

- ❌ 목표 등록 없이 태스크 배분 시작 → 진척 추적 불가
- ❌ 완료 조건 없이 목표 등록 → 언제 끝나는지 모름
- ❌ 이터레이션 로그 미기록 → 다음 세션에서 맥락 소실
- ❌ quality-gate 없이 DONE 처리 → 품질 보장 실패
- ❌ 미완료 항목 발견 후 사용자에게 "계속할까요?" 질문 → 자율 재개해야 함
- ✅ 목표 등록 → 분해 → 루프 → 검증 → 완료 or 재루프
- ✅ 세션 시작 시 pm_progress_guide.md 먼저 읽고 IN_PROGRESS 항목 파악
- ✅ 완료 보고에 잔여 항목 명시
